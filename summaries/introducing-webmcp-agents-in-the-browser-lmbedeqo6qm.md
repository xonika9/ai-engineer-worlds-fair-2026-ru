## 1. О чём доклад

RL Nabors показывает, как с помощью уже существующих браузерных примитивов сделать веб-сайт доступным для трёх типов клиентов: людей в браузере, людей в агентах и агентов, действующих от имени пользователя прямо в браузере. Демонстрируется это на примере переноса архива старых веб-комиксов в агентную среду через MCP‑сервер, MCP‑apps и Web MCP.

## 2. Главные идеи

- MCP‑сервер можно разместить на вебе через HTTP endpoint, что радикально упрощает подключение по сравнению с локальным stdio.
- MCP‑apps — это самодостаточные HTML‑файлы (single‑file), которые встраиваются в интерфейс агента, превращая чат‑стену в интерактивный опыт.
- Web MCP даёт возможность регистрировать инструменты прямо на странице через JavaScript, позволяя встроенному агенту браузера вызывать функции без скриншотов и тяжёлого парсинга DOM.

## 3. Практические выводы

- HTTP‑транспорт удобнее для конечного пользователя: достаточно вставить URL, а не настраивать командную строку.
- MCP‑app нужно собирать как единый HTML, используя бандлинг (V single file), общую дизайн‑систему и курсовую политику (CSP) для внешних ресурсов.
- Внутри MCP‑app ограничены: нет localStorage, ссылки требуют специального метода `appref`, внешние шрифты и картинки блокируются без CSP.
- Инструменты, которые предназначены только для генерации UI‑ответа, стоит помечать `visibility: "app"`, чтобы модель не пыталась интерпретировать их как текстовые данные.
- Web MCP пока не полностью стандартизирован, но его уже можно изучать через специальное браузерное расширение.

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы

MCP (Model Context Protocol), HTTP‑ и stdio‑транспорты, MCP‑apps (single HTML, iframe sandbox), Web MCP (императивная и декларативная модель: `navigator.modelContext.registerTools`), V single file, расширение MCP B, Web Speech API, Web Animations API, Claude, A Rise, Amazon, W3C, Mozilla Firefox DevTools, Microsoft Edge.

## 5. Что полезно разработчику

- Как хостить MCP‑сервер как обычный веб‑сервис (HTTP/SSE, serverless‑функции) и добиться простого подключения.
- Способ упаковки дизайн‑системы в изолированное MCP‑app с учётом жёстких ограничений sandbox’а и CSP.
- Приёмы работы с навигацией внутри app‑а (вызовы back/forward через call‑server‑tool и appref вместо window.open).
- Понимание различий между stdio и HTTP‑транспортом для архитектурных решений.
- Черновой доступ к Web MCP, позволяющий «объяснить» агенту браузера структуру страницы без визуального или DOM‑скрапинга.

## 6. Что можно применить в агентных продуктах

- Вместо стен текста в чате — интерактивные островки UI (MCP‑apps): чтение комиксов, дашборды, формы, работающие внутри агента.
- Раздача контента через MCP‑инструменты, возвращающие структурированный JSON или Markdown (например, транскрипты).
- При наличии поддержки — использование MCP‑resources для предварительной загрузки контекста (документация, справочные данные), чтобы не гонять туда‑сюда инструменты‑«читалки».
- Web MCP для inline‑агентов в браузере: навигация и поиск по странице по прямым вызовам JS‑функций вместо дорогих визуальных моделей.
