## 1. О чём доклад
Дэвид Крамер (Sentry) делится опытом создания MCP-сервера для своего продукта и размышляет, почему протокол пока не готов для серьёзного использования. Фокус — на B2B SaaS и удалённых сценариях, где MCP выступает как подключаемая архитектура для агентов.

## 2. Главные идеи
- MCP — это pluggable architecture для агентов, а не просто обёртка над API.
- Нельзя механически проксировать OpenAPI в инструменты: модели плохо работают с большими JSON-ответами, нужно проектировать выдачу под агента и контекст.
- Агенты — это по сути сервисы с новым названием; реальная ценность лежит в качественных агентах, а не в протоколе.
- Удалённый MCP‑сервер с OAuth 2.1 предпочтительнее для бизнеса: даёт контроль, итеративность и безопасность, в отличие от stdio.
- Безопасность в stdio крайне низкая, случайные MCP‑инструменты опасны.
- У протокола пока нет streaming‑ответов, что усложняет агентные сценарии.
- Важно контролировать стоимость: возврат гигантских payloads резко увеличивает токен‑расход и ломает логику вызовов.

## 3. Практические выводы
- Возвращайте модели Markdown вместо сырого JSON — так она лучше понимает контекст.
- Проектируйте описания инструментов и сообщения об ошибках как человекочитаемый контекст: машина «думает» паттернами языка.
- Не выставляйте наружу всё API; создавайте специализированные, контекстно‑ориентированные инструменты.
- Стройте собственных агентов, которые управляют вызовами и моделью, а через MCP интегрируйте их как плагины.
- Будьте готовы итеративно дорабатывать сервер: это не «сделал и забыл», а еженедельная подстройка под поведение моделей.
- Используйте OAuth 2.1 для удалённых сред; многие клиенты (VS Code, Cursor) уже поддерживают авторизацию.

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы
- Sentry (application monitoring), Sentry MCP server
- VS Code Insiders (OAuth‑поддержка), Cursor, Cloud Code
- Cloudflare Workers (OAuth‑прокси, worker‑shim для OAuth 2.1)
- MCP remote package
- OAuth 2.1, стандартный ввод‑вывод (stdio), WebSocket

## 5. Что полезно разработчику
- MCP‑сервер можно построить за пару дней, низкий порог входа.
- Ответы в Markdown снижают когнитивную нагрузку модели и повышают надёжность.
- Ошибки инструментов должны быть информативными, как для человека, — модель ищет в них подсказки.
- Помните о лимитах токенов в описаниях инструментов; урезайте всё лишнее.
- Активно тестируйте и обновляйте сервер, ориентируясь на реальное поведение LLM.
- Если нет своей WebSocket‑инфраструктуры, используйте Cloudflare Workers или аналоги.

## 6. Что можно применить в агентных продуктах
- Выносите интеллект в собственного агента: он контролирует промпты, модель, объединяет несколько вызовов и снижает зависимость от внешнего клиента.
- Интеграцию с редакторами и другими средами делайте через MCP как плагин‑слой, а не как единственный интерфейс.
- При отсутствии потоковой выдачи (streaming) используйте polling: запуск задачи и последующая проверка статуса, хотя агент может «сдаться» раньше времени.
