## 1. О чём доклад  
Доклад объясняет, почему традиционные CI/CD-пайплайны не справляются с агентной разработкой, и описывает переход к архитектуре *continuous compute*, где код пишут агенты, сборка и тесты встроены во внутренний цикл, а внешнюю валидацию выполняют другие агенты без участия человека.

## 2. Главные идеи  
- Агенты генерируют тысячи короткоживущих веток и PR; слияние в одном репозитории превращается в проблему сериализации, похожую на блокировки в высоконагруженных базах данных.  
- Модель «человек-ревьювер» больше не масштабируется – люди не успевают просматривать в 4 раза больше PR.  
- Предлагается цикл без пул-реквестов: *intent/plan → agent harness (сборка, тесты, внутренняя валидация) → внешняя валидация агентами безопасности и API → pre-merge очередь для согласования параллельных изменений → человеческое утверждение результата, а не кода*.  
- В перспективе – работа в «мультивселенной»: агенты могут стартовать с нескольких кандидатных коммитов одновременно, что ещё сильнее требует высокой скорости внутреннего цикла и инкрементальных вычислений.  
- Кэш становится слоем оркестрации, решающую роль играет совместное проектирование аппаратного и программного обеспечения.

## 3. Практические выводы  
- Инварианты CI (известный чекаут, отсутствие регрессий, соответствие требованиям) должны проверяться непрерывно внутри агентного цикла, а не в отдельной фазе.  
- Нужно отказаться от запуска сборок и тестов «с нуля» – агентам требуются состояние и тёплые кэши.  
- Валидацию кода должны выполнять специализированные агенты (безопасность, соответствие API), а человек – утверждать результат на уровне *намерение/результат*.  
- Для сериализуемости изменений необходима pre-merge очередь, которая согласует параллельные правки до вливания в основную ветку.

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы  
- **Namespace** (высокопроизводительная инфраструктура для непрерывных вычислений).  
- **GitHub Actions, GitHub, Copilot**.  
- **amp, cursor, factory** – harness/среды для агентной разработки.  
- **Linear, Slack** – фиксация intent/плана.  
- **HashiCorp** – упомянут Mitchell Hashimoto с идеями по перестройке GitHub под нужды инференса.  
- Компании **Fall, Zed, Ramp** как примеры тех, кто уже движется к описанной модели.

## 5. Что полезно разработчику  
- Понимание, что пул-реквесты и ручные ревью становятся бутылочным горлышком при агентной генерации кода.  
- Подход к организации цикла: спецификация, контур агента, быстрые внутренние проверки, автоматическая внешняя валидация, утверждение человеком только итогов.  
- Идея инкрементальных сборок и stateful-окружений снижает задержки и избыточную работу.

## 6. Что можно применить в агентных продуктах  
- Встроить валидацию (тесты, сборку, анализ безопасности) непосредственно в агентный цикл, избавившись от ручных PR.  
- Использовать pre-merge очередь для сериализации параллельных изменений при высокой частоте коммитов.  
- Передать внешнюю оценку кода другим агентам-специалистам, сохранив для человека только финальное «принять/отклонить» по намерению и факту.
