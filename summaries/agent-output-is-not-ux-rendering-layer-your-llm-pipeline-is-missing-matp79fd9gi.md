## 1. О чём доклад
Спикер из Amazon (Bala Ramdoss) показывает, что между моделью/агентом и экраном пользователя нужен специальный слой — **Generative UI**. Он решает проблемы доставки: как превратить сырой вывод модели в нативный, интерактивный и отзывчивый интерфейс, не ломая клиент при обновлениях.

## 2. Главные идеи
*Вывод агента — это не UX; вы строите поверх него.*  
- Три ключевых паттерна: **rendering‑контракт** (версионированный контекст, модель выбирает из фиксированного набора компонентов), **стриминг** (потоковая отрисовка по частям, time to first chunk вместо полной задержки) и **BFF** (сервер‑управляемый UI, который гидратирует данные и привязывает действия, логирование и контекст диалога).
- Мобильные клиенты нельзя пропатчить мгновенно, поэтому клиент должен оставаться «тупым» и безопасно деградировать.
- Спектр Generative UI: от выбора готовых компонентов до открытой генерации (A2UI от Google, Co‑pilot Kit).

## 3. Практические выводы
- Передавайте модели версионированный контракт с меню доступных UI‑блоков и правилами компоновки (например, 1-3 рейса — карусель, ≥4 — вертикальный список).
- Перейдите от спиннера к потоковой доставке: показывайте скелетон, затем частично заполненный компонент, потом финальный результат. Измеряйте *time to first chunk*.
- BFF (Backend for Frontend) должен перерабатывать вывод модели в платформенные нативные компоненты с уже привязанными действиями, метриками и контекстом беседы. Это позволяет переиспользовать существующие UI‑юниты (продуктовые карточки, ряды рейсов).

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы
- **A2UI** (Google) — открытая спецификация для описания UI как данных
- **Co‑pilot Kit** — фреймворк для генеративного UI
- **Amazon Lens** — пример продукта с AI-камерой
- **Server‑driven UI / BFF** — подход к серверному управлению отрисовкой
- **Generative UI** (спектр: контроль, декларативность, открытая генерация)

## 5. Что полезно разработчику
Доклад напрямую не касается перечисленных тем, но подход к слою между моделью и интерфейсом критичен для любых агентов: **браузерные/офисные агенты**, использующие tool execution, могут выиграть от стриминга прогресса и рендеринга UI‑компонентов вместо сырого текста. Контракт с моделью и BFF применимы в observability и evals, когда нужно собирать структурированные логи действий.

## 6. Что можно применить в агентных продуктах
- Внедрить **версионированный контракт** для агентов: выдавать модели только те UI‑компоненты, которые поддерживает текущая сборка студии.
- Использовать **потоковую отрисовку блоков** (progress → skeleton → filled UI) вместо ожидания полного ответа.
- Построить **BFF‑прослойку**, которая превращает вывод агента в готовые action‑payloads, привязывает логирование и контекст диалога, сохраняя нативный look & feel студии.
