## 1. О чём доклад
Доклад об опыте Zapier в построении надёжных оценок (eval) для AI‑агентов и создании цикла улучшения на основе данных. За два года разработки платформы для нетехнических пользователей спикеры убедились: *«…users are even more nondeterministic. They are going to use your products in a way that you cannot imagine up front.»*

## 2. Главные идеи
- Вероятностное ПО требует «маховика данных»: прототип → сбор сбоев → создание эвалов → улучшение → рост пользователей.
- Иерархия эвалов: юнит‑тесты (отлов конкретных сбоев), траекторные эвалы (полная цепочка действий), LLM‑судья с рубриками (обзор возможностей), A/B‑тесты (финальная проверка на пользователях).
- Юнит‑тесты склонны переобучаться под старые модели, из‑за чего более сильные новые модели показывают худшие результаты.
- Метрики не должны быть самоцелью: *«when a good metrics become a target, it ceases to be a good target».*

## 3. Практические выводы
- Собирать неявную обратную связь: включение агента, копирование ответа, фрустрация в чате, нецензурная лексика, отток клиентов.
- Детально инструментировать код: трассировать вызовы инструментов, ошибки, пре‑/постобработку; хранить данные в форме, удобной для превращения трассы в eval.
- Зеркалировать пользовательские окружения (не моки), чтобы траекторные эвалы отражали реальность.
- Применять reasoning‑модели для автоматического объяснения корневых причин сбоев и сравнения поведения разных LLM.
- Сделать инстинктом однокликовое превращение любого сбоя в eval.
- Разделить датасет на регрессионный и аспирационный, чтобы не гнаться за 100% метрикой.
- Финальный арбитр — A/B‑тест на малой доле трафика с контролем активации и удержания.

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы
- Компания и продукт: Zapier, Zapier Agents.
- Библиотеки: LangChain (blankchain).
- Платформы: Braintrust, Model Context Protocol (MCP).
- Инструменты разработки: Cursor, Cloud Cod.
- Модели: Gemini Pro, Claude, reasoning‑модели.
- Подходы: data flywheel, пирамида тестирования (unit test evals, trajectory evals, LLM‑as‑judge, rubric‑based scoring, A/B testing).

## 5. Что полезно разработчику
- Не полагаться только на явную обратную связь — неявные сигналы гораздо богаче.
- Подход «один клик — eval» ускоряет накопление тестов и делает анализ систематическим.
- Траекторные эвалы и рубрики позволяют честно сравнивать модели, не привязываясь к единственному пути.
- Reasoning‑модели ускоряют поиск корневых причин по трассам.
- Лабораторные метрики — не истина; реальное качество проверяется A/B‑тестом.

## 6. Что можно применить в агентных продуктах
- Пирамида эвалов: юнит‑тесты для регрессий, траекторные для многошаговых сценариев, LLM‑судья с рубриками для глобальной оценки, контролируемые A/B‑развёртывания.
- Автоматическое выявление фрустрации через LLM и еженедельные отчёты.
- Регрессионный и аспирационный датасеты для защиты текущих сценариев и стимулирования прогресса.
- Зеркалирование пользовательских сред без моков для честных траекторных тестов.
