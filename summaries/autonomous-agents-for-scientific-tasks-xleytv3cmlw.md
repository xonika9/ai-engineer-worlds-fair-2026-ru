## 1. О чём доклад

Сина Шахандех из Radicait показывает, как заставить кодинг-агента не упираться в потолок на открытых научных задачах. Проблема: агенты хорошо реализуют код и гоняют эксперименты, но у них заканчиваются идеи — не хватает «research taste». Его приём — явно разложить задачу на иерархию компонентов (данные, архитектура, лосс, метрики) и на этом каркасе генерировать десятки гипотез, включая радикальные, которые агент сам бы не придумал. Всё на живом примере Radicait: генерация PET-скана из CT для поиска опухолей в лёгких, с петлями оптимизации в Codex и внешними моделями (Gemini, GPT-5.5 Pro) для ревью и критики.

## 2. Главные идеи

- Кодинг-агенты сильны в реализации, но на открытых, длинных научных задачах выходят на плато — им не хватает не рук, а гипотез («research taste»); хорошие исследователи в этой точке продолжают расти.
- Память об ошибках и саму реализацию агенты уже более-менее закрывают; настоящее узкое место — генерация хорошей гипотезы.
- Научную задачу раскладывают на шаги, и каждый шаг — это цикл: наблюдение → гипотеза → реализация → эксперимент → вывод → повтор (hill-climb).
- Ключевой приём: явная декомпозиция задачи на иерархию компонентов (данные, архитектура, training loss, метрики, вспомогательные скрипты) — как chain-of-thought, но для структуры проблемы; иерархию строит сам агент по кодовой базе.
- На таком каркасе агент генерирует ~100 гипотез, модифицируя каждый компонент, и выдаёт радикальные идеи (например, переход с 2.5D на 3D-свёртки), которые без каркаса просто пропустил бы.
- Петлю усиливают вторыми моделями — и адверсариально, и коллаборативно: одна ревьюит снимки (мультимодальная), другая, сильная в рассуждениях, генерирует и критикует гипотезы.
- Главное узкое место науки — наблюдение: сегодняшние мультимодальные модели не «видят» научных деталей (например, наличие нодуля), потому что не обучены на таких данных.
- Приём с иерархиями — это способ масштабировать test-time compute; по мере улучшения моделей нужда в таких костылях будет падать.

## 3. Практические выводы

- Не проси агента «оптимизируй всё» — сначала заставь его явно разложить проблему на иерархию связанных документов-компонентов.
- Генерируй гипотезы по каждому компоненту отдельно: так всплывают радикальные варианты, а не только перебор гиперпараметров.
- Встрой в петлю вторую модель-критика: на каждой итерации спрашивай «вот что сделали, вот результат — имеет ли это смысл, что дальше».
- Для того, что агент не видит сам (качество снимка, выравнивание CT и PET), делай отдельный skill с мультимодальной моделью как метрику качества.
- Тяжёлые рассуждения выноси на более сильную модель через инструмент, который пакует код и данные и отправляет их в API (пример — Oracle CLI + GPT-5.5 Pro).

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы

- Radicait — задача in-silico PET: генерация PET-скана из CT.
- Andrej Karpathy — исходный репозиторий auto-research (program.md, hill-climb по метрике).
- Codex — прогон циклов оптимизации модели.
- Obsidian — просмотр графа иерархии документов-компонентов.
- GAN, encoder-decoder — модель трансляции CT→PET; image registration — выравнивание CT и PET.
- Gemini — мультимодальное ревью сгенерированных снимков.
- GPT-5 / GPT-5.5 Pro — генерация и критика гипотез; Oracle CLI (в докладе — инструмент Peter Spinberger), пакующий код и данные для отправки в API.
- Chain-of-thought — аналогия для приёма с иерархической декомпозицией.

## 5. Что полезно разработчику

- Готовый приём против «плато» кодинг-агента: явная иерархическая декомпозиция задачи перед оптимизацией.
- Паттерн петли с отдельной моделью-критиком и вопросом «что дальше» на каждой итерации.
- Идея выносить то, что агент не умеет наблюдать, в отдельные skills-метрики (мультимодальный контроль качества).

## 6. Что можно применить в агентных продуктах

- Скаффолд из иерархии компонентов, по которому reasoning-агент проходит и предлагает улучшение каждого узла — более полный поиск решений, чем «оптимизируй всё».
- Адверсариально-коллаборативная петля: worker генерирует, второй агент/модель ревьюит план и критикует результат.
- Подключение внешних сильных моделей через инструмент, пакующий контекст (код + данные) и отправляющий его в API, для тяжёлых рассуждений.
- Skills как метрики качества там, где нужен взгляд, которого нет у основной модели (мультимодальный контроль снимков).

## 7. Стоит ли смотреть полностью: maybe — одна сильная переносимая идея (иерархическая декомпозиция ради генерации гипотез), остальное — прикладная специфика медицинской визуализации.
