## 1. О чём доклад

Обзор рынка данных для обучения frontier-моделей: как индустрия дата-лейблинга «раскручивается» из вертикально-интегрированных гигантов в фрагментированную экосистему специалистов. Спикер утверждает, что данные — недофинансированная нога в связке compute-data-talent, и что настоящую ценность несут не готовые «state-based» строки, а process-based данные (траектории, reasoning-трейсы, последовательности решений). Через «закон верификатора» Джейсона Вэя и три оси верифицируемости он объясняет, почему домены созревают в порядке code → search → finance → healthcare → law и как рынок данных служит опережающим индикатором будущих продуктов лабораторий.

## 2. Главные идеи

- Данные для белых воротничков — как уголь и железо для викторианской эпохи; TAM здесь — весь труд.
- Цепочка поставок необратимо фрагментируется: специалисты обыгрывают гигантов на каждом шаге (поиск людей, среды, награды, evals); лаборатории намеренно диверсифицируют по 20-30 вендорам, не веря в масштабирование качества.
- Type 1 (реальные рабочие процессы — GitHub-коммиты, session replays) против Type 2 (контривные примеры от нанятых экспертов): Type 1 тянет модель с 20% до 80%, потому что реализм унаследован из самой работы; «грязный секрет» — почти все продают Type 2.
- Закон верификатора: обучаемость задаче пропорциональна её верифицируемости. Три оси — асимметрия, достоверность (consensus о «правильном») и обилие свежих примеров верифицированной работы.
- Code созрел первым не случайно: GitHub из web 2.0 дал unit-тесты, консенсус о «рабочем коде» и бесконечные примеры с commit-сообщениями как бесплатными reasoning-трейсами.
- Большинство бенчмарков «тихо фейковые»: закон Гудхарта с мотивом прибыли — вендор пишет бенчмарк и продаёт данные, чтобы по нему же карабкаться; contrived-бенчмарки тестируют изолированный вопрос, а не длинную зависимую цепочку.
- Рынок данных — предиктивный сигнал: траты Anthropic на кибербез-данные (январь) и биоданные (март-апрель) предвосхитили выход соответствующих продуктов через 2-3 месяца.

## 3. Практические выводы

- Не аутсорсите определение «реализма» тем же вендорам, у кого покупаете evals и задачи, — это как позволить автору теста ставить себе оценку.
- Единственная метрика на одном scaffold — это один сэмпл из неизмеренного распределения; делайте кросс-harness и кросс-инфраструктурный бенчмаркинг.
- Оценивайте домены по трём осям верифицируемости, чтобы предсказать, какие созреют следующими, вместо гонки за хайпом.
- Проверяйте long-horizon по жёстким критериям: гетерогенные незаменяемые tool-calls, состояния, реально ограничивающие будущие действия, обязательное восстановление после ошибок.
- Ищите пять сигналов в сырых данных: последовательные решения по одной сущности, выводимое экспертное действие на шаг, исходы, зафиксированные независимыми сторонами, и экономически доступную высокооплачиваемую работу.
- Для билдера ров — не сами данные, а pipeline в реальную работу плюс инфраструктура для переобучения по мере улучшения базовых моделей.

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы

- Дата-компании: Scale AI, Surge, Remarque/Mercor (имена в ASR искажены), упоминание приобретения Scale.
- Концепции: type 1 / type 2 (contrived) data, state-based vs process-based data, «Antikythera mechanisms» (перевод бизнес-контекста в evals), RL-as-a-service, RL environment reports.
- Verifier's Law Джейсона Вэя; GPQA-датасеты; бенчмарки уровня «frontier/deep squeeze» (названия искажены в ASR).
- Модели в примерах: Opus 4.7 vs 4.8, GPT 5.5, GLM 5.2; домены — cyber, bio, science, taste; продукты Anthropic вроде Claude bio / life sciences.

## 5. Что полезно разработчику

Доклад даёт разработчику словарь и рамку для критической оценки бенчмарков и датасетов: понимание разницы type 1/type 2 и трёх осей верифицируемости помогает не доверять красивым лидербордам вслепую. Пример с тремя финансовыми задачами показывает, как rubric-анализ (детерминированные верификаторы + корректно применённый LLM-as-judge) вскрывает, что модели ошибаются в противоположных направлениях — GPT точен в арифметике, Opus в методологии.

## 6. Что можно применить в агентных продуктах

- Строить оценку агентов на process-based данных и длинных зависимых эпизодах, а не на изолированных вопросах.
- Комбинировать детерминированные верификаторы с LLM-as-judge и rubric-анализом для long-horizon неверифицируемых задач.
- Оценивать целевой домен по трём осям верифицируемости, чтобы понять, насколько реалистично добиться надёжного агента.
- Проектировать абстракционный слой поверх сменяемых базовых моделей: маршрутизация мелких моделей по cost/latency, управление RL-датасетами и автоматический перезапуск post-training при смене open-source-базы.

## 7. Стоит ли смотреть полностью

Да — для тех, кто работает с обучающими данными, evals и RL-средами, это плотный «инсайдерский» обзор рынка с полезной аналитической рамкой. Речь местами сбивчива, много незнакомых имён и аббревиатур, поэтому нетехнической аудитории будет тяжеловато.
