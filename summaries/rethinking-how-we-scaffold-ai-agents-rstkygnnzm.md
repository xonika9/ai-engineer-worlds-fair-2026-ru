## 1. О чём доклад
Доклад о том, как переосмыслить архитектуру агентов на базе LLM, опираясь на экспоненциальный рост «интеллекта» моделей. Автор на примере production‑агента в Ramp показывает, что подходы, жёстко фиксирующие логику в классическом коде, проигрывают системам, где большая доля вычислений происходит внутри LLM, а не вокруг неё. Ключевая идея — *the Bitter Lesson*: ставка на масштабируемые вычислениями методы выигрывает у тщательно спроектированных «человеческих» эвристик.

## 2. Главные идеи
- Системы, способные просто использовать больше compute, со временем обходят детерминированные, завязанные на код решения.
- Три уровня архитектуры: жёсткий классический код; классический код, иногда вызывающий LLM; архитектура, перевёрнутая в пользу LLM, — LLM сама решает, когда войти в «классическую землю» через код‑интерпретатор.
- Практический пример: агент парсинга CSV (switching report). Если дать LLM произвольный CSV, код‑интерпретатор и верификатор, и запустить 50 параллельных попыток, надёжность резко возрастает при копеечной стоимости.
- *«Сколько "синих стрелок" (вычислений в нейросети) вы пропускаете через свою систему, столько улучшений вы получаете бесплатно, пока лаборатории продолжают вкладывать миллиарды в модели».*

## 3. Практические выводы
- Выгоднее переплачивать за compute, чем писать/поддерживать жёсткий код для каждого частного случая.
- При сценариях с высокой вариативностью входа (CSV, пользовательские форматы) параллельный запуск LLM с инструментами и верификатором даёт устойчивое качество без ручного маппинга схем.
- Смещение центра тяжести с классического бэкенда в сторону «LLM как бэкенд» — направление, которое будет только усиливаться.

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы
- **Компании/продукты:** Ramp, OpenAI, Gmail (в демо).
- **Библиотеки:** `json-former` (авторская structured‑extraction library), pandas / rust‑based csv‑parser (code interpreter).
- **Подходы:** deterministic flow, constrained agent (LLM вызывается из классического кода), flipped agent (LLM управляет вызовами), embedding‑based semantic similarity, parallel LLM sampling с верификацией.
- **Примеры Bitter Lesson:** chess, Go, компьютерное зрение, Atari‑игры.

## 5. Что полезно разработчику
- Правило «не пиши хитрый код вокруг модели, если можно дать модели более общий инструмент и несколько попыток» экономит инженерное время.
- Паттерн «LLM + code interpreter + verifier + parallel runs» применим к любым задачам парсинга, трансформации данных или генерации кода в production.
- При проектировании агента полезно оценивать долю «fuzzy compute»: чем её больше, тем сильнее система выиграет от следующего поколения моделей без переписывания.

## 6. Что можно применить в агентных продуктах
- Архитектура, где агент сам решает, выйти ли в детерминированный код, а не жёстко прописанный пайплайн с точечными вызовами LLM.
- Механизм множественных параллельных попыток с проверкой результата для критичных операций, где цена ошибки высока.
- Концепция «LLM как бэкенд»: модель получает доступ к данным и инструментам, сама рендерит UI/ответ, а система лишь передаёт события (клики, запросы). Это кардинально снижает объём промежуточного кода.

## 7. Стоит ли смотреть полностью: yes / maybe / no
yes — живое демо и эволюция архитектуры на реальном продукте хорошо иллюстрируют центральный тезис.
