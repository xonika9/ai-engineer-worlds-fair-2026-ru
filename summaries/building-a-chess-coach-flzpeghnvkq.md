## 1. О чём доклад

Доклад о создании AI-тренера по шахматам в приложении Play Magnus (ныне Take Take Take). Описывается пайплайн генерации разбора партий, где Stockfish, тактические детекторы и движок Maya поставляют контекст, а LLM переводит его в комментарий на естественном языке. Также показан автономный агент, замыкающий цикл обратной связи: жалоба пользователя → событие в MCP-канале → автоматическое расследование, правка промптов/детекторов и создание PR.

## 2. Главные идеи

- История шахматных движков от переборных до AlphaZero, и почему LLM плохо играют, но хорошо объясняют.
- Пайплайн обзора партии: движок Stockfish даёт оценку, набор тактических и позиционных детекторов извлекает угрозы, связки, вилки и т.д., нейросетевой движок Maya предсказывает вероятности ходов человека на разных уровнях рейтинга.
- LLM только переводит этот структурированный контекст в текст, не пытаясь «думать» за движок — это уменьшает галлюцинации.
- Замыкание цикла через агента на базе Cloud Code Channel (MCP): даунвот → Slack → событие в канал → агент запускает навык triage, правит контекст/промпт, перегенерирует ответ и при успехе отправляет PR с мобильного.
- Компромисс latency vs. quality: для мгновенного фидбека выбрана Gemini 3 Flash (≈3 сек), более мощные reasoning‑модели (Claude, GPT‑5 mini) медленнее и нестабильнее при сопоставимом качестве.
- Оценка качества (evals): 16 сценариев на тактики и зевки, LLM‑as‑judge, перебор моделей через Open Router; финальное решение — эксперты с хорошим уровнем игры.

## 3. Практические выводы

- Чётко отделяйте извлечение данных (контекст) от генерации текста — LLM работает надёжнее, когда опирается на готовый структурированный анализ.
- Замыкайте обратную связь через автономных агентов — это позволяет быстро итерироваться и вносить улучшения буквально с телефона.
- Контекст начинает жизнь как большой JSON; постепенное прореживание улучшает качество, хотя начальный этап трудоёмок.
- Автоматизированные evals необходимы, но для финальной оценки привлекайте экспертов предметной области (в данном случае сами разработчики — сильные шахматисты).

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы

- **Stockfish** — ведущий шахматный движок.
- **Maya** — исследовательский движок (University of Toronto), предсказывающий ходы человека по рейтингу.
- **Gemini 3 Flash (Google)**, **Claude (Anthropic)**, **GPT‑5 mini (OpenAI)** — использовавшиеся LLM.
- **Open Router** — платформа для быстрой замены моделей при тестировании.
- **Cloud Code Channel (MCP‑сервер)** — постоянная сессия для инжекта событий и запуска агента.
- **Slack** и **GitHub** — для уведомлений и автоматических PR.
- **Kaggle Game Arena** — бенчмарк LLM в шахматах.
- **AlphaZero (DeepMind)** — пример трансформерного движка без языкового обучения, играющего на уровне гроссмейстера.

## 5. Что полезно разработчику

- Метод «разделяй и властвуй»: вынос точных вычислений из LLM снижает галлюцинации и ускоряет ответ.
- Практичный шаблон реактивного агента с MCP‑каналом для автоматического улучшения AI‑фич по пользовательским жалобам.
- Подход к оценке качества: набор сценариев из реальных данных, LLM‑as‑judge, A/B‑тестирование новых моделей через API‑шлюз вроде Open Router.
- Понимание, что быстрое поколение (sub‑3s) требует быстрых моделей; reasoning‑модели оставить для асинхронных сценариев, где пользователь готов ждать.

## 6. Что можно применить в агентных продуктах

- Замыкание петли обратной связи: триггер от пользователя → уведомление в коммуникатор (Slack/Teams) → инжект события в постоянно работающий MCP‑канал → агент запускает специализированный навык, меняет контекст/промпт, перезапускает генерацию, проверяет результат, при необходимости задаёт уточняющие вопросы человеку и автоматически создаёт PR.
- Паттерн безопасного экспериментирования: агент действует в изолированной среде, все изменения проходят через код‑ревью.
- Использование одного канала для непрерывного обслуживания нескольких инцидентов.
