## 1. О чём доклад
Практический воркшоп по интеграции A2A (Agent-to-Agent) и MCP (Model Context Protocol) для автоматизации бизнес-процессов. Строится мультиагентная система: хост-агент оркестрирует суб‑агентов (Slack, GitHub, удалённый Bench), агенты общаются по A2A, а MCP‑инструменты (через Zapier) выполняют конкретные действия. Система запускается по вебхуку с транскриптом встречи.

## 2. Главные идеи
- A2A — для удалённых агентов «без контроля» (агенты третьих сторон), MCP — для подключения внешних инструментов без написания API‑интеграций.
- Если есть полный контроль над функциями и агентами, A2A и MCP избыточны; локальные вызовы проще и быстрее.
- Главное преимущество A2A — изоляция контекста: суб‑агент переваривает большой объём данных и возвращает хосту только сжатый результат, снижая затраты и рост системного промпта.
- Prompt caching сложен: оптимальный порог кэширования (по симуляциям) около 30 тыс. токенов; универсальной стратегии нет, нужен анализ пользовательского поведения.
- MCP‑серверы (особенно Slack от Zapier) могут «молча сбоить», требуя дополнительных проверок.

## 3. Практические выводы
- Для продукта с множеством интеграций можно дать пользователям подключать собственные MCP‑серверы, но на практике их возможностей часто не хватает — приходится строить first‑party интеграции с индексацией и кэшированием.
- Стоит жёстко указывать конкретные инструменты MCP, чтобы агент не вызывал лишнее (например, не постил в случайные каналы).
- Агентская карточка (agent card) должна быть краткой: перегрузка описаний способностей сбивает хост-агента.
- При зависимостях между шагами (создать Issue → отправить в Slack со ссылкой) нужно отключать параллельное выполнение суб‑агентов.

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы
- Google A2A, Genkit, Gemini API.
- MCP, Zapier MCP, MCP Inspector, Asana MCP, Linear MCP.
- Bench (автономный AI‑агент, агрегатор LLM).
- Claude (Opus, Sonnet, Haiku), Gemini Flash/Pro, DeepSeek.
- Slack, GitHub, Salesforce (упоминаются).
- Подходы: изоляция контекста через суб‑агентов, prompt caching, OAuth для скоупинга данных, направленные ациклические графы (DAG) для композитных задач.

## 5. Что полезно разработчику
- При использовании MCP добавляйте проверки: Slack от Zapier возвращает пустой массив при несуществующем канале, не кидая ошибку.
- В Genkit встроено ограничение — максимум 5 вызовов суб‑агентов за ход (баг).
- Связка A2A + MCP в Genkit работает через Genkit XMCP; готовой нативной поддержки пока нет.
- Отладка удалённого A2A агента затруднена — вы не видите его логи.
- Для кэширования промптов проводите симуляции на реальных сессиях; один размер не подходит всем, экономия 25‑35% при риске выбросов.

## 6. Что можно применить в агентных продуктах
- Выделенные суб‑агенты под каждый инструмент, чтобы защитить основной контекст от раздувания.
- Подключение будущих first‑party A2A‑агентов (например, от Salesforce) для доступа к более продвинутым возможностям, чем просто MCP‑инструменты.
- Паттерн «человек как инструмент»: описать роль (CFO) как tool, который через сообщение в Slack запрашивает информацию у живого сотрудника.
- Управление сложными процессами через DAG с fan‑out/fan‑in, аналогично CI/CD.
- OAuth в MCP для автоматического ограничения доступа данными конкретного пользователя.

## 7. Стоит ли смотреть полностью: maybe
Доклад даёт живой пример кода и освещает реальные боли интеграции A2A+MCP. Однако A2A ещё очень сырой, значительную часть времени занимают демо и дискуссия. Смотреть стоит для понимания текущих ограничений и практических лайфхаков, но не за глубокой теорией.
