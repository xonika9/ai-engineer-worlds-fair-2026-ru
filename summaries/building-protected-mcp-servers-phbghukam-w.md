## 1. О чём доклад
Защита удалённых MCP-серверов. Почему важно уметь авторизовывать клиентов, если сервер подключается к защищённым API. Обзор текущей (уже стабильной) и грядущей редакции спецификации авторизации MCP. Демонстрация, как легко добавить OAuth-защиту в MCP-сервер с помощью C# SDK, Azure API Management и стандартных identity-провайдеров.

## 2. Главные идеи
- Локальным MCP-серверам специальная авторизация не нужна: бинарник может использовать любые секреты и сертификаты машины.
- Удалённый сервер обязан проверять контекст пользователя, потому что разные запросы требуют разных прав.  
- Старая (stable) спецификация вынуждала разработчика реализовывать собственный авторизационный сервер (чеканку токенов), что требовало глубоких знаний OAuth.  
- Новый черновик (скоро стабильный) разделяет ресурсный сервер и сервер авторизации. MCP-сервер лишь публикует метаданные защищённого ресурса (PRM), а клиент выполняет стандартный OAuth‑танец.  
- Благодаря этому можно подключать любые Identity Provider’ы (Entra ID, Okta, Keycloak) и полагаться на готовые библиотеки, а не писать код безопасности вручную.

## 3. Практические выводы
- Достаточно добавить несколько строк конфигурации в ASP.NET Core, чтобы включить OAuth-защиту MCP-сервера.  
- Сервер валидирует токен, полученный клиентом; чеканка и управление жизненным циклом токена вынесены на стандартный auth-сервер.  
- Размещать удалённый MCP-сервер безопаснее за шлюзом вроде Azure API Management, который берёт на себя авторизацию и проксирование.  
- VS Code Insiders уже поддерживает новый spec и способен выполнять аутентификацию с показом consent-страницы.

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы
- MCP (спецификации), MCP Inspector  
- C# SDK для MCP, ASP.NET Core  
- Azure API Management, Microsoft Entra ID  
- VS Code Insiders, GitHub Copilot  
- Okta, Auth0, Keycloak, OAuth 2.0, OpenID Connect  
- Protected Resource Metadata (PRM), подход «ресурсный сервер vs. авторизационный сервер»

## 5. Что полезно разработчику
- Не нужно быть экспертом по OAuth: стандартные библиотеки фреймворка (AddAuthentication, MCP OAuth scheme) скрывают сложность.  
- Чёткое разделение ответственности: клиент делает dance, сервер только проверяет токен.  
- Готовый пример на GitHub разворачивает защищённый сервер с API Management через Azure Developer CLI за несколько минут.  
- Код сервера тривиален: задать провайдера, скоупы, метаданные PRM – и защита работает.

## 6. Что можно применить в агентных продуктах
Агентные клиенты (Claude Desktop, VS Code с Copilot) получают безопасный доступ к инструментам удалённого MCP-сервера с учётом прав конкретного пользователя. Унифицированный OAuth-поток позволяет агентам автоматически аутентифицироваться и вызывать защищённые endpoint’ы без ручного управления токенами.
