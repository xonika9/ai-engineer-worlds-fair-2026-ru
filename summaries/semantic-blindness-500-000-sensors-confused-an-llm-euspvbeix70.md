## 1. О чём доклад

Доклад о «семантической слепоте» — ситуации, когда LLM путается в сотнях тысяч похожих названий датчиков ЦОД. Традиционные методы (загрузка имён в контекст, RAG, шардирование) ломались из‑за штрафа частоты и малой разницы между строками. Phaidra предложила решение, которое растёт не с количеством устройств, а с глубиной иерархического дерева.

## 2. Главные идеи

- Промышленные объекты образуют дерево (дата-холл→ряд→стойка→GPU); глубина растёт медленно, ширина — взрывно.
- Четыре инсайта:
  - **Linearizer**: описать все возможные пути к устройствам коротким списком, делая контекст почти постоянным.
  - **LLM — планировщик, а не поисковик**: модель выдаёт структурированный план (scope/поддерево/фильтр).
  - **Детерминированный резольвер**: предварительно проиндексированные поддеревья + операции над множествами дают 100% точность.
  - **Шаблоны вместо перебора**: для нечётких запросов LLM выдаёт шаблон имён, а поиск выполняется в коде.
- Архитектура: запрос → planner LLM → структурированный поисковый план → deterministic resolver → точный result set.
- Инверсия Software 3.0→1.0 (по Карпати): AI-native продукты рождаются как гибкий промт, но по мере масштабирования критичные, структурированные части переходят в детерминированный код, оставляя LLM интерпретацию намерений и синтез ответа.

## 3. Практические выводы

- Не заставляйте LLM перебирать имена; превратите её в генератор плана поиска.
- Используйте иерархическую структуру и компактный linearizer, чтобы контекстное окно не росло с масштабом.
- Детерминированные операции (индексы, пересечения множеств) обеспечивают воспроизводимость и постоянную стоимость запроса.
- Начинайте с чистого LLM-прототипа (Software 3.0), но переносите предсказуемые части в код (Software 1.0), сохраняя модель для сложных решений. *«Develop with software 3.0, продакшенize by adding in software 1.0»*.
- Тщательное тестирование на реальных данных: 66 кейсов на 6 промышленных системах показали нуль ошибок.

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы

- Phaidra (создатели AI-агентов для ЦОД).
- Неудавшиеся подходы: наивная загрузка имён в контекст, RAG, векторный поиск, параллельные вызовы с шардированием.
- Рабочее решение: linearizer, planner LLM со структурированным выводом, deterministic resolver, pre-indexed subtrees, операции над множествами, поиск по шаблонам имён.
- Концепция Software 1.0 / 3.0 А. Карпати.

## 5. Что полезно разработчику

- Паттерн «планировщик + детерминированный резольвер» идеален для навигации по большим DOM‑деревьям: LLM выдаёт scope и фильтр, а поиск выполняется через XPath/селекторы без сканирования тысяч элементов.
- Linearizer можно использовать для сжатого описания UI‑иерархии в промте, экономя токены.
- Строгий eval-конвейер (замер точности и стоимости на реальных кейсах) — прямой рецепт для observability и тестирования агентов в агентном продукте.
- Инструменты песочниц и выполнения кода выигрывают от выноса точных операций в детерминированные модули, оставляя LLM роль диспетчера планов.

## 6. Что можно применить в агентных продуктах

- Создать компонент Linearizer для иерархических сущностей (UI‑дерево, файловая система, документы) и подавать его в системный промт агента.
- Реализовать связку «LLM‑планировщик → structured output {entity_type, scope_path, filter} → Deterministic Resolver» для точного поиска элементов без галлюцинаций.
- Для офисных агентов: шаблоны поиска («все таблицы с заголовком X в разделе Y») вместо отправки полного документа в LLM.
- Внедрить конвейер сравнения старого и нового подхода по точности и затратам токенов.
