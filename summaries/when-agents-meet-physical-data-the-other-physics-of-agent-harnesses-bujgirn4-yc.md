## 1. О чём доклад
Доклад о проблемах работы AI-агентов с неструктурированными «физическими» данными — видео, сенсоры, телеметрия роботов. Спикер показывает, почему обычные подходы проваливаются, и предлагает архитектуру «упряжи для данных» (*data harness*), которая помогает кодинг-агентам эффективно понимать, обрабатывать и переиспользовать результаты работы с бинарными файлами.

## 2. Главные идеи
- Агенты плохи в работе с данными без дополнительного контекста: Anthropic показали точность 21%, OpenAI описали 6 слоёв контекста.
- Неструктурированные данные похожи на «нейтронную звезду»: небольшой объём на входе (тысячи файлов) взрывается в миллионы внутренних объектов.
- Хранение метаинформации в JSON на S3 или в отдельной базе SQL порождает «острова» и безумную сложность. Выход — строгие Pydantic-схемы и автоматическая трансляция в SQL.
- Упряжь для данных должна давать агенту возможность *«видеть, ходить, трогать и запоминать данные»*: схематизация, движок выполнения, инкрементальные чекпоинты, тесты и переиспользуемая база знаний.
- Результаты обработки нужно сохранять в дата-сеты со связанным исходным кодом, чтобы не перевычислять одно и то же многократно.

## 3. Практические выводы
- Не складывать метаинформацию в JSON-файлы рядом с бинарниками — это ведёт к чудовищной латентности и несогласованности.
- Использовать Python/Pydantic как единый язык для схем и кода, избегая SQL-«островов».
- При обработке больших объёмов физических данных обязателен движок с параллельным/распределённым выполнением и чекпоинтами — иначе потеря вычислений при сбое неизбежна.
- Строить многоуровневую аналитическую модель (star schema, one big table) поверх сырых данных, чтобы отвечать на вопросы быстро, без повторного запуска тяжёлых вычислений.
- Делиться дата-сетами, контекстом, исходным кодом в виде простой базы знаний (MD-файлы) — это устраняет дублирование дорогих LLM-прогонов.

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы
- **DataChain** (опенсорс-проект для data harness), **Pydantic**, **YOLO**.
- Исследования **Anthropic** (21% точности агентов на данных) и **OpenAI** (6 слоёв контекста для data-агентов).
- **Dask** как пример подхода к соединению вычислений и структурированных данных.
- Кодинг-агенты: **Cloud Code**, **Copilot**, **Codex**.
- Подходы: инкрементальная обработка, чекпоинты, многомерная модель данных (star schema), MCP-подобное расширение контекста.

## 5. Что полезно разработчику: AI-агенты, браузерные и офисные агенты, harness, sandboxes, tool execution, developer tools, evals, observability, агентном продукте
- Принцип «упряжи» как прослойки между агентом и неструктурированным хранилищем прямо применим к браузерным/офисным агентам, работающим с файлами и веб-контентом.
- Подход с Pydantic-схемами и автогенерацией метаданных можно использовать в tool execution для структурирования результатов работы sandbox-инструментов.
- Концепция чекпоинтов и инкрементальных обновлений критична для evals и observability при прогоне агентов на больших тестовых наборах.
- Лёгкая база знаний из MD-файлов с исходным кодом и контекстом — паттерн для организации памяти агента и дешёвого переиспользования результатов.

## 6. Что можно применить в агентных продуктах
- Строить автоматическую генерацию структурированных дата-сетов (Pydantic) поверх обработанных пользовательских файлов, чтобы избегать тяжёлых повторных вычислений.
- Внедрить чекпоинты для длинных агентных цепочек, чтобы сбои не обнуляли прогресс.
- Создавать для каждого дата-сета MD-файл с контекстом, схемой и сгенерированным исходным кодом — это даст прозрачность, воспроизводимость и возможность шаринга между агентами в студии.
- Использовать лёгкий локальный движок выполнения (map/generator) как основу для распределённой обработки в sandbox-окружениях.

## 7. Возможная идея для поста
Использовать метафору «нейтронной звезды данных», чтобы объяснить, почему «простая папка с картинками» — это на самом деле миллионы скрытых связей, и как data harness превращает эту массу в понятную агенту материю. Хороший визуальный крючок для поста в соцсетях.
