## 1. О чём доклад
Как применять OAuth для безопасного подключения AI-агентов и MCP-серверов, разбор текущих проблем спецификации MCP и обзор будущих требований к авторизации в мире агентов.

## 2. Главные идеи
- Полезность агентов растёт с числом подключённых систем, но сегодня приходится выбирать: *“we can give agents broad‑based access and accept security risks or we can limit their capabilities and sacrifice business value.”*
- Массовое использование долгоживущих API-ключей в конфигурациях и переменных окружения создаёт серьёзную проблему безопасности.
- OAuth разделяет роли (клиент, ресурсный сервер, сервер авторизации) и даёт динамический доступ вместо статических секретов.
- Текущая спецификация MCP (конец марта) некорректно сливает роли ресурсного сервера и сервера авторизации в одно целое, что признано ошибкой сообществом; ожидается исправление в следующем драфте.
- Для безопасного будущего агентов необходимы: аутентификация агентов, транзакционная авторизация, цепочка доверия (chain of custody), асинхронные и real‑time взаимодействия.

## 3. Практические выводы
- Не внедряйте авторизацию по текущему черновику MCP — дождитесь чёткого разделения, где MCP-сервер выступает только как resource server.
- Для межсерверных взаимодействий агентов используйте client credentials flow.
- При идентификации агентов: для публичных клиентов — push client registration (упрощённая модель), для проверяемых — URL + PKI с подписью JWT или HTTP Message Signatures.
- В цепочках вызовов применяйте OAuth token exchange и identity assertion grant, чтобы сохранять контекст авторизации между доменами.
- Для тонкой настройки прав в чувствительных транзакциях изучите Rich Authorization Requests (лимиты, бюджеты, конкретные операции).
- Учитывайте аттестацию среды выполнения агента, чтобы знать, какой LLM получает данные, и включайте это в процесс авторизации.
- При удалённых и фоновых агентах готовьте асинхронные OAuth-флоу через SMS, push или real‑time протоколы (SIP, XMPP, WebRTC).

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы

- OAuth token exchange, identity assertion grant

## 5. Что полезно разработчику
- Чёткое понимание трёх ролей OAuth и того, как MCP-сервер должен реализовывать только роль resource server.
- Отказ от ручной регистрации клиентов в пользу push client registration для публичных агентов.
- Криптографически верифицируемая идентичность агента на основе URL.
- Схема передачи контекста авторизации между серверами через обмен токенами.
- Расширение OAuth на фоновые и голосовые сценарии с использованием push-уведомлений и существующих real‑time протоколов.

## 6. Что можно применить в агентных продуктах
- Разделение аутентификации и авторизации: OpenID Connect + OAuth.
- Динамическую регистрацию и проверку агентов с помощью PKI.
- Транзакционные разрешения с бюджетами и уровнями доступа (RAR).
- Сквозную цепочку доверия от агента до нижележащих API.
- Аттестацию окружения для контроля попадания данных в LLM.
