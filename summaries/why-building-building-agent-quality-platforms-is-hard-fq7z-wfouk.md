## 1. О чём доклад  
Доклад о поэтапном построении платформ для оценки качества агентов (agent quality platforms). На примере BrainTrust разбирается, почему задача выходит далеко за рамки простых скриптов и таблиц, превращаясь в сложную системную проблему, охватывающую observability, эксперименты и анализ реальных данных.

## 2. Главные идеи  
- LLM крайне вариативны, агенты становятся нормой — без системных evals высоки риски для бренда, compliance и стоимости обслуживания.  
- Старт с электронных таблиц и for-циклов допустим, но это лишь *документирование*, а не итеративное улучшение.  
- Зрелая платформа объединяет observability и evals в единый цикл: продакшен-трейсы → офлайн-эксперименты → улучшение агента.  
- Трейсы агентов огромны, слабоструктурированы и требуют особых подходов к хранению (полнотекстовый поиск, низкая задержка, агрегация).  
- Платформа должна обслуживать не только людей, но и coding-агентов (headless-сценарии), уметь выявлять «неизвестные неизвестные» через тематическое моделирование.

## 3. Практические выводы  
- Начинайте с простого, но быстро переходите к настоящим экспериментам, иначе застрянете в документировании.  
- Вовлекайте domain-экспертов и нетехнических пользователей — создавайте для них удобные «песочницы».  
- Свяжите observability и evals: используйте реальные продакшен-трейсы для формирования офлайн-тестов.  
- При самостоятельной разработке платформы будьте готовы управлять ей как продуктом и решать проблемы производительности данных.  
- Автоматизируйте сбор трейсов через AI-прокси/шлюз, чтобы исключить пропуски.  
- Предусмотрите ролевое управление доступом и маскирование данных при масштабировании.

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы  
- BrainTrust  
- Neon, Postgres, DuckDB  
- Open-source data warehouse  
- Braintrust BTQL (собственный язык, признанный неудачным)  
- AI proxy/gateway для автоматического трейсинга  
- Подходы: offline/online evals, алертинг, полнотекстовый поиск по трейсам, тематическое моделирование, coding agents (Codex/Cloud Code)

## 5. Что полезно разработчику  
- Понимание, что evals — непрерывный процесс, а не разовая проверка.  
- Архитектурные инсайты: трейсы требуют минимум двух паттернов запросов — низколатентного для UI и аналитического для агрегации; мультимодальные данные хранить в object storage с отображением в интерфейсе.  
- Предупреждение: «vibe-coding» платформы приведёт к тому, что её поддержка станет основной работой.  
- Идея замкнутого цикла: production traces → offline evals → улучшение.

## 6. Что можно применить в агентных продуктах  
- Внедрите цикл observability–evals: собирайте трафик из продакшена, анализируйте неудачи, улучшайте промпты и параметры в «песочнице», снова выкатывайте.  
- Поставьте онлайн-оценки и алертинг для проактивного выявления проблем.  
- Используйте автоматический трейсинг через прокси для сплошного покрытия.  
- Применяйте тематическое моделирование на трейсах, чтобы находить неочевидные паттерны использования и планировать доработки.  
- Проектируйте eval-платформу так, чтобы с ней могли взаимодействовать агенты (headless API).
