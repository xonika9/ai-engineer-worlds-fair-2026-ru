## 1. О чём доклад
Доклад посвящён «массовому психозу» вокруг агентов – ощущению, что все вокруг строят роботов, а вы не понимаете, с чего начать. Ара Хан предлагает четыре уровня зрелости, чтобы осмысленно создавать полезных AI-агентов: от быстрой проверки гипотезы до продакшен-масштабирования. Цель – замедлиться и выстроить понятную дорожную карту.

## 2. Главные идеи
- **Четыре уровня зрелости агентов**:  
  1) Использовать готовый фреймворк, чтобы быстро понять, работает ли идея.  
  2) Писать агента самостоятельно как стейт-машину.  
  3) Перейти к UX-форм-фактору Kanban-досок, становясь «инженерным менеджером» для параллельных агентов.  
  4) Вынести исполнение в облако для масштабирования и долгих задач.
- **5 правил при самостоятельной сборке**: агент – это стейт-машина; любое новое условие рискует ухудшить результат; делайте агента частью RL-пайплайна с удобным CLI; не плодите «слоп» – продумывайте архитектуру; Frontier Labs намеренно усложняют заменяемость через API.

## 3. Практические выводы
- Для проверки гипотезы хватит 30 минут с LangChain/LangGraph; не застревайте на этапе фреймворков.
- При серьёзной разработке стройте агента как явный конечный автомат с понятными переходами.
- Минимизируйте системные промпты: *«передовые модели настолько хороши, что чем меньше инструкций, тем лучше они работают»*.
- Организуйте работу с агентом через CLI, чтобы другие кодинг-агенты могли легко дорабатывать и тестировать вашего агента.
- Используйте Kanban-доски, чтобы запускать нескольких агентов параллельно и не ждать завершения каждого.
- Запускайте агентов в облаке для изолированного и масштабируемого выполнения, особенно для задач длительностью 50–60 минут.

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы
- Фреймворки: LangChain, LangGraph.
- Продукты/интерфейсы: Codex, Cursor, Factory, Cline, Clock Code client, Linear.
- Провайдеры моделей: Frontier Labs, Open Router.

- IDE/среды: VS Code.

## 5. Что полезно разработчику
- Пять правил «второго уровня» дают конкретные ориентиры для написания собственного агента.
- Явное осознание, что каждый добавленный `if` или edge case может сделать поведение модели хуже, помогает избегать переусложнения.
- Приём с CLI и `agents.md`: проектируйте агента так, чтобы другой кодинг-агент мог автоматически собирать, тестировать и дорабатывать его.
- Предупреждение о форматах `reasoning traces`: неправильная передача снижает производительность, и вы можете этого не заметить.

## 6. Что можно применить в агентных продуктах
- Внедрить Kanban как основной UX для оркестрации нескольких агентов; пользователь видит заголовки задач и может переключаться между потоками.
- Облачное выполнение как стандартный шаг для продакшена: изолированная среда, отсутствие локальных зависимостей, параллельный запуск множества инстансов.
- Итеративный переход по уровням зрелости в зависимости от стадии продукта.
