## 1. О чём доклад

Кейнот команды OpenAI (Romain Huet, Alexander Embiricos, гость Peter Steinberger) о том, почему сейчас «золотой век» инженерии, а не её конец. Тезис: инженерия всегда была про решение задач, а не про написание кода, поэтому усиление моделей не убирает инженеров, а возвращает профессию к её корням. Спикеры показывают, как быстро ускоряется цикл выпуска моделей и продуктов, разбирают открытую многослойную архитектуру Codex (модель → Responses API → harness → app server → приложение/плагины) и вводят идею «value maxing» — извлечения ценности из агентов через стоимость, скорость и параллельный запуск в изоляции. Финальная часть Питера Штайнбергера — про переход от «жонглирования 20 терминалами» к управлению «менеджером» агентов во внешнем цикле.

## 2. Главные идеи

- Инженерия — это решение проблем с помощью науки, вкуса, суждения и воображения; код всегда был лишь средством, поэтому «AI-инженеры съедают мир».
- Темп ускорился: раньше новую модель выпускали примерно раз в 15 месяцев, теперь — примерно раз в 6 недель; анонсирован preview семейства 5.6.
- Эволюция агентов: автодополнение → inline-предсказание → команда на правку (Command K) → модели, тестирующие свою работу → модели, доводящие длинные сложные цели до конца.
- Агент может делать всё, что человек делает на своём компьютере: помогает не только с кодом, но и с тем, что до (зачем нужно) и после (ревью, деплой) — так замыкается «петля».
- Продуктовая цель — не автоматизировать инженеров, а максимально их усилить; удобная форма — две модальности: chat (недооценён) и мощный совместный UI для «погружения в детали».
- Codex построен как открытые слои, которыми пользуется сама OpenAI: тот же Responses API, open-source harness, AGENTS.md, open-source app server, расширяемые примитивы и плагины.
- «Value maxing» вместо «token maxing»: важны стоимость (эффективность), скорость (например, интеллект фронтира на Cerebras ~750 ток/с) и параллельный запуск многих изолированных задач.
- Будущее — без искусственного разделения «локальная задача / облачная задача»: агент сам выбирает окружение.
- Штайнбергер: модели развиваются быстрее, чем harness и организации вокруг них; проектирование этих «петель» — следующая инженерная задача.

## 3. Практические выводы

- Строить агентов не как «наблюдение из-за плеча», а как делегирование: talk-first, углубляться в детали только когда нужно.
- Замыкать цикл: связывать агента с контекстом «зачем» до работы и с ревью/деплоем после — так он начинает и «доводит» больше работы.
- Использовать общие примитивы API (например, compaction для длинных контекстов), а не изобретать своё — то, что OpenAI строит для себя, попадает в API.
- Форкать и адаптировать open-source harness/app server, использовать AGENTS.md как общий формат инструкций; при желании подключать открытые модели, сохраняя тот же agent loop.
- Использовать скорость не для одного быстрого ответа, а для параллельных подходов (5–6 вариантов) с выбором лучшего.
- Переходить от роли «планировщика/роутера/памяти» вручную к менеджеру-агенту: сервер-сайд compaction, координация, триггеры (persistent context + delegation + triggers = «петля»).
- Определять, где тратить внимание (главный дефицит) — не тратить его на постоянное «рулёжку» модели, которая уже хорошо понимает намерение.

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы

- OpenAI, Codex (Codex app, Codex Cloud, VS Code extension, Codex Monitor, Codex for iOS), goal mode, remote.
- Модели семейства 5.6: GPT 5.6 (варианты Sol/Sole, Terra, Luna по расшифровке), а также упоминания GPT 5.3, GPT 5.5, o1 (в контексте Dev Day 2024).
- Responses API, compaction, open-source harness, AGENTS.md, app server, плагины (browser use, computer use, ролевые плагины для data science/дизайна).
- Экосистема, использующая подписку: Open Code, Pi, Droids, Open Claw, Xcode, JetBrains.
- Cerebras (инференс на высокой скорости), «test boxes» для запуска тестов отдельно.
- Подходы: value maxing vs token maxing, inner loop / outer loop, менеджер агентов.

## 5. Что полезно разработчику

Доклад задаёт продуктовую и ментальную рамку для работы с агентами: не «сидеть над агентом», а делегировать и погружаться в детали по необходимости. Практически ценно то, что OpenAI отдаёт те же примитивы, что использует сама: Responses API, открытый harness, app server, AGENTS.md — на этом можно строить собственные инструменты и нативные приложения (пример — Codex Monitor, собранный до официального приложения). Полезен и разбор «движущегося бутылочного горлышка»: сначала токены, потом compute (решается вынесением тестов на отдельные машины), затем — внимание человека.

## 6. Что можно применить в агентных продуктах

- Проектировать две модальности: chat как основной интерфейс и совместный UI для инспекции/управления, сохраняя у пользователя чувство «мастерства» над работой.
- Замыкать полную петлю задачи: контекст «зачем» → выполнение → ревью/деплой, а не только генерация кода.
- Опираться на общие примитивы (compaction, единый формат инструкций AGENTS.md), а не на закрытые адаптеры; делать точки расширения (плагины) публичными.
- Строить архитектуру «менеджер + воркеры»: долгоживущий менеджер-агент делегирует задачи, будится по триггерам (issue, событие), возвращает человеку PR/дифф/видео/сборку для ревью во внешнем цикле.
- Абстрагировать место запуска: агент сам решает, что выполнить в облаке, а что требует локальной машины; давать возможность управлять им из Slack/сообщений «откуда угодно».

## 7. Стоит ли смотреть полностью

Да, если интересны продуктовое видение OpenAI по агентам и открытая архитектура Codex. Кейнот вдохновляющий и обзорный (много маркетинга и анонсов, мало технических деталей), но финальная часть Штайнбергера про переход к «менеджеру агентов» и внешнему циклу — концентрированная и практичная.
