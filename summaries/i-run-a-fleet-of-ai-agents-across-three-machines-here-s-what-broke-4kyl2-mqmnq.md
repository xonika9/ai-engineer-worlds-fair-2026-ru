## 1. О чём доклад

Практический опыт запуска персонального флота AI-агентов на трёх машинах, поиск архитектуры, которая убирает человека из роли планировщика, памяти и рецензента.

## 2. Главные идеи

- Человек не должен держать в голове контекст всех агентов. Нужна иерархия: CEO, VP, manager, worker с разделением информации.
- Состояние живёт в файлах (рабочие директории с `mission`, `status`, `handoff`), а не только в контекстном окне модели.
- Вместо дорогой компактификации — полный сброс контекста и восстановление из handoff-файлов.
- Единая точка ревью (web-инбокс) для утверждения планов всех слоёв.
- Масштабирование на несколько машин: Git для передачи контекста, единый шлюз ревью на always-on Linux‑машине, Discord как мобильный роутер.
- На будущее — оркестрация по образцу Kubernetes: отделение логической иерархии от физического размещения задач.

## 3. Практические выводы

- Стройте иерархию агентов с ограниченным контекстом на каждом уровне.
- Храните состояние в файлах, чтобы работа переживала сбои и перезагрузки.
- Используйте полный сброс контекста с чтением handoff вместо автоматических саммари.
- Внедрите единую точку ревью — это резко снижает когнитивную нагрузку.
- Для нескольких машин разделите длинные и короткие задачи, синхронизируйте контекст через Git с pull request для общих данных, перенесите контроль на always-on машину.
- CLI harness принудительно заставляет агента делегировать задачи, если он норовит всё делать сам.

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы

tmux, tmux capture-pane/send-keys, SSH, Git, pull request, Claude, MCP-процессы, Discord, Kubernetes. Подходы — иерархия агентов, file-based state, context reset вместо compact, review gateway, per-machine workspace, overlord boot.

## 5. Что полезно разработчику

- **AI-агенты** — центральная тема; **harness** — упомянут CLI harness для делегирования; **sandboxes** — изоляция workspace и разделение креденшелов; **tool execution** — CLI и MCP-процессы; **developer tools** — агенты сами построили review gateway; **evals и observability** — прямых упоминаний нет, observability частично реализована через централизованный инбокс и Discord-уведомления. Браузерные/офисные агенты и агентный продукт не затрагиваются.

## 6. Что можно применить в агентных продуктах

Иерархическую организацию агентов с передачей контекста через файлы, механизм ручного ревью планов (gateway) перед выполнением действий, хранение состояния на диске для отказоустойчивости, CLI harness как способ гарантированного делегирования, Git‑синхронизацию контекста при работе на нескольких машинах.
