## 1. О чём доклад

Fireside-разговор с Cat Wu и Thariq Shihipar (Anthropic) о том, как за полтора года изменилась работа с Claude Code — от эпохи, когда приходилось вычитывать каждый permission-запрос, до auto mode и нового командного продукта Claude Tag (Claude, живущий прямо в Slack). Спикеры разбирают, как внутри Anthropic принимают решения о фичах через dogfooding, как устроены evals и автоматический code review (вплоть до удаления человека из петли на внешних слоях), как auto mode защищает от prompt injection через sonnet-классификатор и sandbox, и почему системный промпт Claude Code удалось сократить на 80%. Сквозная мысль: ценность инженера смещается от исполнения к product taste, амбиции и вкусу.

## 2. Главные идеи

- Работа с агентом прошла путь от ручного контроля каждого шага к делегированию рутины; новое поколение моделей (упоминается модель под кодовым названием Fable) нередко решает фичу за один проход (one-shot).
- Ценность инженера сместилась: меньше акцента на исполнение, больше — на product sense и business sense, «что вообще стоит строить», раз путь от идеи до реализации сжался с полугода-года до недели.
- Rewrites перестали быть табу: при хорошем тест-сьюте переписывание кода снова разумно; кодовая база сама по себе — это спецификация, иногда единственная её копия.
- Claude Tag — эволюция Claude Code для команд: multiplayer по умолчанию, проактивный (мониторит баг-репорты в канале и сам заводит PR), с командной памятью (markdown-файл на канал). Внутри Anthropic Claude Tag закрывает ~65% продуктовых PR.
- Приоритизация фич решается через жёсткую dogfooding-культуру: у фичи есть внутренняя планка по активным пользователям и retention до релиза наружу.
- Code review: для критического ядра — обязательный code owner и ручное ревью; для внешних слоёв Claude ревьюит полностью. Это результат 6+ месяцев наращивания доверия и eval-инфраструктуры; PR из инцидентов добавляют в eval-наборы против регрессий.
- Evals — фундамент: новая модель проходит весь набор и должна быть строго лучше предыдущей, чтобы стать drop-in заменой. Есть capability-evals и отдельные behavioral-evals (чтобы Claude не говорил «пора спать» или не спрашивал «продолжать ли»).
- Сокращение системного промпта на 80%: сильные модели переставали нуждаться в примерах (примеры даже ограничивали креативность), и вместо жёстких «do not» лучше давать больше контекста и меньше конфликтующих с инструкциями пользователя запретов. Для разных моделей теперь разные системные промпты.
- Промпт должен быть «100% верным», потому что подаётся модели в 100% случаев: инструкцию «always verify» смягчили, полагаясь на судейство модели (что доступно только фронтир-моделям уровня Opus/Fable).
- Auto mode — рекомендованный внутри Anthropic способ безопасной длинной работы: sonnet-классификатор оценивает каждый tool/bash-вызов в контексте, поддерживает динамические разрешения из промпта и работает с сетевым sandbox. Тысячи evals и red-team; заявляют, что закрыли практически все найденные атаки (не 100%, но риск ниже среднего человека-ревьюера).
- Тренд по инструментам — меньше tools с чёткими непересекающимися функциями (убрали grep/glob в пользу нативного bash); file edit оставили в основном ради детерминированного UI подтверждения правок.

## 3. Практические выводы

- Развивайте product taste и бизнес-чутьё — цикл «идея → продукт» сжался, и решает выбор «что строить», а не скорость набора кода.
- При хорошем тест-сьюте не бойтесь переписывать: можно поднять 3 реализации против одного eval-набора и выбрать лучшую; больше прототипируйте.
- Проектируя промпт, ищите edge-кейсы: если утверждение верно на 90%, но есть реальные 10% исключений — смягчайте формулировку, чтобы не сбивать модель.
- Давайте моделям контекст, а не гору жёстких запретов и примеров; «do not»-инструкции особенно конфликтуют с пользовательскими и путают модель.
- Для команд рассматривайте многопользовательский, проактивный агент в коллаборационных инструментах (по образцу Claude Tag) вместо самописного AI-Slackbot — у самодельного слишком много attack-векторов.
- Стройте доверие к автоматическому ревью постепенно: начинайте с ручного ревью везде, фиксируйте, где авто-ревью ловит 100% проблем, и убирайте человека только там; из каждого инцидента добавляйте PR в eval.
- Держите каталог tools маленьким: каждый инструмент должен иметь функцию, отличимую от других, чтобы модель понимала, когда его звать.

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы

- Claude Code, Claude Tag (Claude в Slack), co-work, workflows, remote control, trusted devices — продукты и фичи Anthropic.
- Модели: Claude Sonnet 3.7, Opus 4, Opus 4.8, а также модель под кодовым названием «Fable» (в расшифровке ASR; упоминаются также «probes», «Mythos» в контексте safety — названия переданы осторожно).
- Auto mode, YOLO mode, sandboxing, sonnet-классификатор разрешений, credential injection (проксирование токенов для доступа к внешним сервисам вроде Datadog).
- Sub-agents, task tool, ask user question tool, file edit tool; отказ от grep/glob в пользу нативного bash.
- Evals (capability и behavioral), red-teaming, prompt injection, data exfiltration, «Swiss cheese» защита, RL против атак.
- Dogfooding / «ant-fooding», code owners, CI/CD, artifacts для объяснения PR.
- Внешние примеры: переписывание Bun на Rust (Jared/Zig упоминаются как история), Gemini API и «sea dance»-модель для генерации анимаций, ffmpeg, remotion, Mountain Project, Airbnb — как кейсы использования.

## 5. Что полезно разработчику

- Понимание, что «узкое место» смещается от написания кода к верификации, вкусу и выбору задач: качество и амбиции важнее объёма.
- Конкретная механика построения доверия к моделям и авто-ревью через evals и постепенное удаление человека из петли — переносимый паттерн для любой команды.
- Практические уроки по промпт-инжинирингу: смягчение абсолютных инструкций, отказ от примеров для сильных моделей, разные промпты под разные модели.
- Идея credential injection (агент может использовать секрет, но не видит его) — полезный безопасный паттерн доступа к внешним API.

## 6. Что можно применить в агентных продуктах

- Проектируйте агента как проактивного участника команды (мониторинг каналов, автозаведение PR, командная память), а не как реактивный чат.
- Стройте eval-инфраструктуру так, чтобы новая модель была drop-in заменой: весь набор прогоняется, новая версия должна быть строго лучше; добавляйте инциденты в eval против регрессий.
- Для безопасности длинных автономных прогонов используйте классификатор действий поверх контекста + sandbox + динамические разрешения из пользовательского запроса; проверяйте red-team’ом.
- Минимизируйте набор инструментов и делайте их функции непересекающимися; давайте моделям контекст вместо жёстких запретов.
- Применяйте credential injection и «Claude как отдельная идентичность» для аудита и ограничения полномочий агента.

## 7. Стоит ли смотреть полностью

Да, если вам интересна внутренняя кухня Anthropic: как устроены dogfooding, evals, auto mode, code review и промпт-дизайн Claude Code. Формат — живой fireside без слайдов, много конкретики о принятии решений и безопасности агентов; ближе к концу — вдохновляющая часть про роль инженера и амбиции.
