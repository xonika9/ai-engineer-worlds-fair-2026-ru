## 1. О чём доклад
Доклад посвящён построению масштабируемого enterprise-уровня RAG в юридической сфере. На примере Harvey AI (ассистент для юрфирм) рассматриваются проблемы поиска по огромным, сложным и конфиденциальным документам, а также инфраструктурные решения на базе LanceDB и AI-native lakehouse.

## 2. Главные идеи
- Сложность юридических запросов требует комбинации семантического, точного (keyword) поиска, фильтрации по датам, нормативным актам и доменным сокращениям.
- Документы очень длинные, с плотным контентом; объёмы достигают десятков миллионов документов.
- Ключ к качеству — eval-driven разработка: от дорогих экспертных ревью до автоматических метрик (precision/recall).
- Инфраструктура должна быть одновременно надёжной для онлайн-обслуживания и гибкой для офлайн-экспериментов.
- AI-native lakehouse (LanceDB) позволяет хранить все типы данных (векторы, текст, бинарные объекты) в одном месте на object store, разделяя compute и storage.
- Lance формат даёт быстрый random access для поиска, поддерживает schema evolution и эффективно смешивает большие BLOB-данные со скалярными значениями.

## 3. Практические выводы
- Инвестируйте в систему оценок разной глубины, чтобы быстро итерировать.
- Тесно работайте с доменными экспертами для понимания структуры данных и неявных паттернов запросов.
- Выбирайте инфраструктуру, которая даёт единое хранилище для всех AI-артефактов, поддерживает тяжёлые batch- и лёгкие online-нагрузки.
- Отдавайте предпочтение форматам с хорошей совместимостью (Apache Arrow) и возможностью легко масштабироваться через Spark/Ray.

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы
- Harvey AI
- LanceDB, Lance формат (AI-native multimodal lakehouse)
- Apache Arrow, Spark, Ray, PyTorch
- pandas, Polars
- GPU-индексирование векторов (рекорд: ~3-4 млрд векторов за 2-3 часа)
- Подходы: разделение compute/storage на object store, комбинированный поиск (векторный + полнотекстовый + реранкинг), оценочный спектр (экспертные ревью, размеченные критерии, автоматические метрики)

## 5. Что полезно разработчику
- Стройте RAG с прицелом на скорость итераций; меняющимся моделям и контекстным окнам нужна быстрая пересборка.
- Используйте lakehouse-архитектуру, чтобы избежать разрозненных копий данных и проблем с синхронизацией.
- Комбинируйте плотный поиск с точным совпадением и фильтрацией, а не полагайтесь только на embeddings.
- Для масштабных данных подключайте GPU-индексирование в один проход по всей таблице.

## 6. Что можно применить в агентных продуктах
- Гибкое retrieval-API, позволяющее динамически совмещать векторный и полнотекстовый поиск с переранжированием, идеально для агентных маршрутов, где запросы сложны и многосоставны.
- Единое хранилище с поддержкой мультимодальных данных (текст, векторы, метаданные) упрощает агентам доступ к любому контексту без дополнительной оркестрации.
- Надёжная оценочная инфраструктура ускоряет добавление новых источников и тактик поиска в агентные цепочки.
