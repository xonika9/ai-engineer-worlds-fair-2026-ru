## 1. О чём доклад  
Доклад о внедрении MCP в масштабах большой организации (Anthropic). Спикер делится опытом построения центрального MCP-шлюза, который унифицирует интеграции, решает проблемы безопасности и аутентификации, и предлагает инженерам максимально простой способ подключения к инструментам моделей.

## 2. Главные идеи  
- Лавина кастомных endpoint’ов приводит к «интеграционному хаосу», а со временем всё равно начинает напоминать MCP.  
- MCP — это прежде всего потоки JSON-RPC сообщений; транспорт (HTTP, WebSockets, gRPC, Unix-сокеты) — деталь реализации.  
- «Pit of success»: правильное действие должно быть самым лёгким. Если дать разработчикам одну функцию `connect_to_mcp`, возвращающую готовый SDK-объект, они естественно пойдут по стандартному пути.  
- Единый MCP Gateway обеспечивает сквозную аутентификацию, rate limiting, observability и портативность учётных данных между сервисами.  
- Централизация всего потока контекста даёт точку аудита и контроля безопасности (защита от prompt injection, блокировка вредоносных серверов).

## 3. Практические выводы  
- Стандартизируйтесь на чём-то (предпочтительно MCP) для передачи контекста моделям — это снижает хаос.  
- Gateway должен брать на себя OAuth‑потоки и хранение токенов, чтобы прикладной код не дублировал эту логику.  
- Стройте клиентские обёртки, возвращающие MCP SDK-сессию; тогда обновления протокола автоматически распространяются на все продукты.  
- Транспорт внутри компании выбирайте под свои нужды (WebSockets, gRPC, Unix-сокеты). Важно на концах передавать read/write streams в SDK.  
- Используйте Gateway для аудита, фильтрации контента и блокировки опасных действий.

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы  
- MCP (Model Context Protocol), JSON-RPC, OAuth 2.1, Streamable HTTP, WebSockets, gRPC, Unix-сокеты, IMAP (ироничный пример).  
- Anthropic, MCP SDK, MCP Gateway.  
- «Pit of success» (яма успеха) — подход, при котором самый лёгкий путь ведёт к правильному результату.

## 5. Что полезно разработчику  
- Не писать каждый раз транспорт и аутентификацию для интеграций — получать их из инфраструктуры через одну точку входа.  
- Переиспользовать одну и ту же интеграцию в разных продуктах и языках (Gateway скрывает разнообразие).  
- Бесплатно получать новые возможности протокола при обновлении SDK‑пакетов.  
- Меньше времени на сантехнику — больше на продуктовую логику.

## 6. Что можно применить в агентных продуктах  
- Единообразное подключение агентов к любым MCP‑серверам (внутренним и внешним) через общий шлюз.  
- Агенты могут действовать от имени пользователя без повторных OAuth‑танцев — credentials привязаны к внутреннему ID.  
- Сквозной трейсинг и аудит всех действий агентов, упрощающий расследования и политики безопасности.  
- Использование MCP-примитивов (например, sampling) для унификации биллинга и ограничений в разных агентных продуктах.
