## 1. О чём доклад

Доклад о том, как переложить скучные, но важные задачи (написание тестов, улучшение читаемости кода, поиск уязвимостей) с разработчика на CI-процесс. Предлагается концепция «ghost pilot» — медленной, обдуманной AI-системы, которая в отличие от быстрого co-pilot’а итеративно анализирует код в pull request’ах, повышая качество и безопасность без потери «состояния потока».

## 2. Главные идеи

- **Co-pilot vs. Ghost Pilot**: Co-pilot — быстрое мышление (Thinking Fast), ghost pilot — медленное, deliberate мышление (Thinking Slow), подходящее для code review и анализа безопасности.
- **Четырёхшаговый CI-пайплайн**:
  1. Улучшение имён переменных и комментариев кода для лучшего понимания моделями.
  2. Итеративная генерация unit-тестов с adversarial mindset, персонализацией и запуском до прохождения.
  3. Поиск логических уязвимостей и нарушений best practices путём симуляции трёх AI-сотрудников (red team, разработчик, engineering manager) с дебатами и саморефлексией.
  4. Предложение исправлений с объяснением приоритетов, привязкой к политикам и прогоном тестов.
- **Контекст решает**: Локальный контекст (context.yaml) и выводы AI о языке, исторических багах, фокусных областях значительно повышают точность анализа.

## 3. Практические выводы

- Итеративный подход с «медленной» LLM в CI даёт более качественные результаты, чем одноразовый запрос.
- Улучшение имён переменных (rev → annual_revenue) и комментариев позволяет модели привлекать широкие предметные знания, выявляя краевые случаи.
- Adversarial mindset (поиск граничных условий, weird characters) расширяет покрытие тестов.
- Симуляция мультиагентных дебатов находит логические дыры, невидимые для статического анализа.
- Автоматические исправления нельзя применять вслепую — нужен human review с готовыми объяснениями.

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы

- GitHub Copilot, Sourcegraph Cody (примеры co-pilot’ов)
- Термин «ghost pilot» (авторский)
- context.yaml (способ хранения метаданных проекта)
- Jira (исторические баги), Slack, Zoom-транскрипты, PRD
- opencontext.org (возможная интеграция для контекста)
- SAST-инструменты (противопоставляются логическому анализу)
- GPT-3.5 (тестировался на реальном баге Kubernetes)
- GitLab CI, GitHub Actions (реализация)

## 5. Что полезно разработчику

- Рутинные задачи (тесты, комментарии, поиск типовых уязвимостей) переносятся в CI, позволяя дольше оставаться в продуктивном «потоке».
- Код получает более осмысленные имена и комментарии, что упрощает дальнейшую поддержку.
- PR-процесс обогащается готовыми предложениями по исправлениям с краткой мотивацией и ссылками на политики безопасности.

## 6. Что можно применить в агентных продуктах

- Мультиагентную симуляцию с распределением ролей (red team, владелец кода, менеджер) и итерационными дебатами для анализа решений.
- Слой контекста в виде подключаемого YAML/базы знаний (аналог context.yaml).
- Итеративную генерацию и верификацию (цикл «тесты → доработка → отражение»).
- Подход «ghost pilot» — медленный, обдуманный агент для задач, где не нужен мгновенный ответ, но важна глубина.

## 7. Стоит ли смотреть полностью: maybe

Доклад-концепция без готового продукта, содержит интересные идеи и практический пример поиска бага, однако многие детали остаются на уровне прототипа. Полезен для тех, кто проектирует AI-ассистентов в CI/CD.
