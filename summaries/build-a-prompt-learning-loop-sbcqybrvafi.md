## 1. О чём доклад
Спикеры из Arize рассказывают о построении цикла обучения промптов (Prompt Learning Loop) для повышения надёжности агентов. Они разбирают, почему агенты ненадёжны, вводят концепцию улучшения системных промптов через итеративную обратную связь (человеческие аннотации и объяснения от LLM-судей), демонстрируют прирост метрик на примере coding agent и проводят практический воркшоп по JSON-промптам.

## 2. Главные идеи
- Агенты часто ломаются не из-за слабости моделей, а из-за плохих инструкций, отсутствия планирования, нехватки инструментов и слабого контекстного инжиниринга.
- Prompt learning заимствует идеи из reinforcement learning и meta-prompting, но ключевое отличие – использование именно текстовых объяснений ошибок (human feedback или объяснения eval-ов), а не только скалярных оценок.
- Системный промпт можно значительно улучшить, добавляя предметные правила на основе размеченных примеров: кейс с coding agent показал рост на 15% без fine-tuning и смены архитектуры.
- Важно оптимизировать не только агентные промпты, но и сами eval-промпты, потому что цикл улучшения опирается на их качество.
- «Переобучение» под конкретный датасет – это скорее экспертиза, а не недостаток, если используется train/test split.

## 3. Практические выводы
- Начинайте с простого прототипа и готовых eval-ов, не давая им блокировать разработку.
- Собирайте датасет из неудачных кейсов с развёрнутой обратной связью (почему ответ неверен, какие правила нарушены).
- Итеративно прогоняйте оптимизацию промпта: генерация на train-выборке, оценка, передача feedback в оптимизатор, повторение до достижения порога качества.
- Используйте асинхронность для ускорения экспериментов и настраивайте параметры (число циклов, размер выборки, количество правил).
- Сохраняйте все результаты экспериментов, чтобы потом выбрать лучшую версию промпта.

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы
- **Arize** (платформа): Prompt Learning SDK, Prompt Hub, задачи оптимизации промптов.
- **OpenAI API** (в воркшопе – GPT-4o-mini).
- **LLM as a judge** (шаблоны evaluator-ов для классификации correct/incorrect и проверки правил).
- **JSON response format** и temperature=0 для репрезентативных результатов.
- **Train/test split** для предотвращения нежелательного переобучения.
- Упомянуты **Claude Code** (как пример coding agent), **GEA** (генетические алгоритмы для оптимизации промптов), **Booking.com** (как клиент с субъективными оценками).
- **Асинхронный подход** (nest_asyncio) для ускорения прогона.

## 5. Что полезно разработчику
- Шаблон построения автоматизированного цикла улучшения промптов с минимальными зависимостями.
- Конкретный пример с JSON-промптом, который можно адаптировать под свои задачи.
- Принцип: обогащайте данные для оптимизатора не только бинарными метками, но и объяснениями.
- Возможность встроить подобный цикл в CI/CD или непрерывно работающую задачу на продуктовых данных.

## 6. Что можно применить в агентных продуктах
- Непрерывное обучение агента на реальных ошибках: сбор трасс, разметка экспертами или eval-ами, автоматическая регенерация системного промпта.
- Два со‑эволюционирующих цикла: улучшение поведения агента и параллельное улучшение самих оценивающих промптов.
- Специализация агентов под конкретную кодовую базу или домен за счёт «экспертного» переобучения на рабочих данных.
