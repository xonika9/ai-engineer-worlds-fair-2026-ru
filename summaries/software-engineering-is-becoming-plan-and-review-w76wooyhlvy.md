- Спикеры: Louis Knight-Webb, Vibe Kanban
- Трек: Coding Agents @ AI Engineer

## 1. О чём доклад  
Как развитие AI-агентов для кодинга превращает работу разработчика в непрерывное планирование и ревью, и как меняется роль человека, когда агенты выполняются всё дольше.

## 2. Главные идеи  
- Работа инженера сместилась: время, которое раньше уходило на написание кода, теперь тратится на планирование и ревью.  
- Два подхода: plan-based (детальный план заранее → меньше ревью) и review-based (быстрый старт → много итераций). Plan-based экономит время.  
- Тип задачи влияет на подход: frontend-фичи трудно специфицировать (лучше оставаться в цикле), backend и миграции можно вести plan-heavy через TDD.  
- Агенты начинают работать дольше (уже 5–10 минут, в перспективе час), что требует параллельного запуска нескольких задач и новых интерфейсов для «фокус-максинга» человека.

## 3. Практические выводы  
- *5 минут планирования экономят 30 минут ревью AI-кода.*  
- Для backend, рефакторинга и миграций используйте test-driven development — это естественная форма plan-based подхода.  
- Когда агент выполняется >5 минут, не ждите — запускайте несколько параллельных задач и переключайтесь между ними по готовности.  
- Интерфейсы должны минимизировать переключение контекста: агент должен работать долго и отдавать результат только тогда, когда нужен человек.

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы  
- GitHub Copilot, ChatGPT, Cursor, Claude Code, Playwright MCP, Codex (упомянут как CLI-агент)  
- Vibe Kanban (стартап спикера для параллельного запуска агентов)  
- Подходы: test-driven development (TDD), spec frameworks, план-документы (markdown)  
- Облачная платформа: Cloudflare CDN

## 5. Что полезно разработчику  
- Осознать, что написание кода перестаёт быть центральной активностью; вы теперь «менеджер агентов».  
- Вкладывать усилия в чёткое планирование задач — это окупается меньшим объёмом ревью.  
- Научиться параллелить работу агентов, чтобы не простаивать во время длинных прогонов.  
- Не прыгать между задачами слишком часто — это снижает продуктивность.

## 6. Что можно применить в агентных продуктах  
- Проектировать интерфейсы, поддерживающие одновременное ведение нескольких задач и минимизирующие переключение контекста.  
- Встраивать в агентов этапы QA (например, автотесты и визуальную проверку через Playwright) для снижения нагрузки на ревью.  
- Строить баланс «планирование vs ревью» прямо в продукте, помогая пользователю тратить время эффективнее.

## 7. Стоит ли смотреть полностью: yes / maybe / no  
yes — неожиданный финал с закрытием стартапа и полезная рамка для осмысления сдвига роли разработчика.
