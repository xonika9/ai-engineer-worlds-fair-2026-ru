## 1. О чём доклад  
Доклад о том, как в incident.io используют AI для управления сложностью собственных AI-продуктов (в первую очередь SRE-расследований). Основной фокус — внутренние инструменты отладки, которые помогают разработчикам и AI-агентам понимать поведение многоагентных систем, улучшать промпты и масштабировать анализ качества.

## 2. Главные идеи  
- Evals (наборы тестов для промптов) со временем становятся громоздкими, и AI-агенты не могут эффективно с ними работать напрямую.  
- Цикл «красный-зелёный» через evals позволяет менять промпты, но не указывает, какую часть большой системы нужно править.  
- Перевод UI-инструментов отладки в самодокументированную файловую систему (дамп всех промптов, трасс, инструментов) резко повышает способность агентов (Claude Code, Codex) находить корневые причины ошибок.  
- Повторяемый анализ тысяч взаимодействий строится через пайплайны с параллельными саб-агентами, кластеризацией отказов и привязкой к кодовой базе, чтобы не просто видеть метрику, а понимать, *почему* система ведёт себя так, и предлагать конкретные исправления.

*«We could have put an MCP on top of this… It wouldn’t have been half as effective as this ability to just download in bulk all of the information».*

## 3. Практические выводы  
- Инструменты отладки нужно проектировать так, чтобы с ними могли эффективно работать как люди, так и AI-агенты.  
- Файловые системы — исключительно удобный контекст для агентов; дампы трасс и промптов работают лучше, чем попытки «живого» доступа через MCP.  
- Любой сложный повторяющийся анализ стоит оформлять как AI runbook — это экономит дни и недели ручной работы.  
- Используйте саб-агентов для параллельного разбора сущностей и сохраняйте промежуточные результаты в файлы, чтобы анализ можно было приостановить и возобновить.

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы  
- **incident.io** — платформа управления инцидентами  
- **Claude Code**, **Codex** (упомянут как Code X) — AI-агенты для работы с кодом и файловыми системами  
- **eval tool** (собственный CLI) — утилита для чтения и редактирования eval-файлов агентами  
- **YAML**-файлы для хранения evals  
- **backtests** — ежедневный пакетный прогон расследований на реальных аккаунтах  
- **runbooks / skills** — инструкции для агентов, описывающие повторяемый процесс  
- **sandbox** — изолированная среда, куда выгружается файловая система для анализа

## 5. Что полезно разработчику  
- Конкретный шаблон: как сделать eval-данные доступными для AI-агентов и замкнуть цикл «найти проблему → исправить промпт → проверить, что не сломано остальное».  
- Приём превращения UI-информации в файловую структуру, позволяющий агенту grep-ом искать нужные детали.  
- Архитектура пайплайна анализа: параллельные агенты → кластеризация → соотнесение с исходным кодом → готовый PR с исправлением.

## 6. Что можно применить в агентных продуктах  
- Методику дампа внутреннего состояния агентной цепочки в файлы для последующего «разбора полётов» силами другого агента.  
- Пайплайн оценки качества с группировкой типовых сбоев и автоматическим формированием рекомендаций по изменению кода.  
- Подход, при котором инструменты для разработчиков и инструменты для агентов проектируются по одним и тем же принципам (удобный текстовый контекст, самодокументированность).

## 8. Самые важные таймкоды  
- [0:02:10–0:03:00] Проблема: чтобы оценить качество одного расследования, нужно вручную разобраться в инциденте, что занимает около часа.  
- [0:06:56–0:08:10] Создание CLI eval tool, позволившего агентам работать с eval-файлами и выполнять цикл «красный-зелёный» по runbook.  
- [0:10:25–0:11:30] Идея загружать всю UI-отладку как файловую систему в sandbox Claude Code; агент сам находит, где ошибка.  
- [0:13:19–0:14:40] Пайплайн анализа: параллельные саб-агенты на каждое расследование, затем кластеризация, интеграция с кодом и предложение исправлений.  
- [0:15:48–0:16:45] Ключевые рекомендации: делайте инструменты удобными для агентов, используйте файловые системы вместо MCP, создавайте AI runbooks для анализа.
