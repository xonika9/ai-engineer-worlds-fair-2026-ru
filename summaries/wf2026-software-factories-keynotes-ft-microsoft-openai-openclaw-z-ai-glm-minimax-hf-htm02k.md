## 1. О чём доклад
Пленарные доклады AI Engineer World's Fair 2026, посвящённые треку Software Factories. Рассматривается переход от единичных агентов к полностью автоматизированным конвейерам разработки ПО — от идеи до мониторинга в продакшне. Сквозная тема — «петли» (loops) и оркестрация агентов, а также экономика токенов, открытые модели, безопасность и инженерная дисциплина при внедрении агентных систем.

## 2. Главные идеи
*   **Эволюция петель:** AI-инженер работает с многоуровневыми петлями — от автодополнения к автономным циклам «задача → спецификация → код → ревью → деплой → мониторинг → улучшение».
*   **Software factory ≠ просто кодинг.** Это полный жизненный цикл: сбор идей, триаж, написание спек, реализация агентами, валидация (включая «прожарку» в песочнице), ревью и непрерывное самоулучшение.
*   **Контроль качества обязателен.** Модели пока не могут поддерживать долгосрочное качество кодовой базы без чтения кода человеком и тщательного архитектурного планирования. «Harness engineering is not enough».
*   **Модели управляют моделями.** Появились агенты-менеджеры, распределяющие работу между суб‑агентами (OpenAI, Anti‑gravity).
*   **Открытые модели снижают риски.** GLM 5.2, MiniMax M3 и другие open‑weight модели дают leverage в переговорах с провайдерами и удешевляют «не‑фронтирные» задачи.
*   **Безопасность агентов.** «Смертельная трифекта» (приватные данные + недоверенный контент + инструменты) требует human‑in‑the‑loop, песочниц и доказательной верификации (Lean, proof‑carrying code).

## 3. Практические выводы
*   Стройте фабрику как конвейер: `issue → triage → spec → implement → review → sandbox verify → monitor → feedback loop`.  
*   Внедряйте **контрольные петли** (control loops) для инкрементального улучшения кода: сенсор (линтер/агент) → контроллер → актуатор с малым PR, а не гигантские слепые правки.  
*   Всегда планируйте архитектуру и дизайн до реализации — это сокращает болезненное ревью и предотвращает «слоп».  
*   Делайте модель‑агностическую архитектуру: подбирайте модель под сложность задачи, используйте open‑weight для рутинных операций.  
*   Для критических систем обязательно изолируйте исполнение в **sandboxes**, добавьте валидаторов‑агентов и утверждение человеком.

## 4. Упомянутые инструменты, фреймворки, компании и подходы
**Компании/лабы:** Microsoft (Foundry, Microsoft IQ), OpenAI (Codex app, GPT‑5.6), Z.ai/GLM 5.2, MiniMax M3, Google (Anti‑gravity), Anthropic (Claude Code), Cursor, Factory.com, Human Layer, Warp, Gravile, Nori Agentic, Resonate, TurboPuffer, OpenGov.  
**Инструменты/фреймворки:** Codex harness (open source), Zcode, Effect (TypeScript‑библиотека для агентных циклов), A2A‑протокол, MCP, sidecars, агентные skills, облачные sandboxes, Chronicle (replayability), Lean для доказательной безопасности.

## 5. Что полезно разработчику
*   **AI-агенты:** оркестрация команд агентов (Anti‑gravity), паттерн «агент‑менеджер» (Codex).  
*   **Браузерные/офисные агенты:** ускорение через сжатое представление страницы и планирование длинных последовательностей (доклад про browser challenge).  
*   **Harness:** эффект‑нативный агентный цикл (Effect), открытый Codex harness.  
*   **Sandboxes:** облачные песочницы для мобильной разработки и изоляции (доклад про 10X).  
*   **Tool execution:** deferred context engine, A2A для контрактов между агентами, sidecars для долгоживущих триггеров.  
*   **Developer tools:** Codex app, Warp, Conductor, Human Layer.  
*   **Evals & observability:** переход от бенчмарков к продакшен‑telemetry, Chronicle для replayability.  
*   **агентном продукте:** непосредственно не упоминается; полезны принципы фабрики, контрольных петель и облачных sandbox‑окружений.

## 6. Что можно применить в агентных продуктах
*   Внедрить фабричный конвейер с этапами `triage → spec → реализация → ревью → sandbox‑верификация`.  
*   Использовать **контрольные циклы** для инкрементальной миграции кодовой базы и автоматического исправления анти‑паттернов.  
*   Генерировать слайды и документы через HTML‑агентов (как у Nori), интегрируя их в студийный процесс.  
*   Подключить внешние модели через MCP/A2A для гибкости и избежания вендор‑лока.

## 7. Возможная идея для поста
«Почему software factories ломаются без чтения кода». На основе конференции: модели хорошо справляются с зелёным полем, но в сложных проектах агенты деградируют качество; как контрольные петли и архитектурное планирование спасают ситуацию. Можно снабдить примерами из доклада Human Layer.

## 8. Стоит ли смотреть полностью
**yes** — материал насыщен практическими кейсами, инженерными деталями и откровенными провалами, что делает его исключительно полезным для всех, кто строит агентные системы.
