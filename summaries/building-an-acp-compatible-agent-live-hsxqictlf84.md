## 1. О чём доклад

Bennet Fenner из Zed за ~15 минут вживую превращает минимального кодинг-агента на TypeScript в агента, совместимого с Agent Client Protocol (ACP) — открытым JSON-RPC-протоколом, который Zed сделал, чтобы любой CLI-агент (Claude Code, Codex, Gemini CLI и другие) работал в редакторе через единый интерфейс, по духу близкий к MCP и LSP. По ходу демо он реализует обязательные функции протокола, добавляет стриминг ответов модели, показ вызовов инструментов и проксирование файловой системы, а под конец агент дописывает сам себе инструмент терминала. Демо идёт с живыми сбоями — дублирующимися событиями и обрывами соединения, которые спикер не успевает чинить.

## 2. Главные идеи

- ACP решает ту же задачу, что LSP для языков: один протокол вместо кучи попарных интеграций, чтобы клиент (редактор) и любой агент говорили через общий интерфейс. Он открытый, на JSON-RPC, работает поверх стандартного ввода-вывода.
- Экосистема уже большая: агенты подключаются напрямую или через адаптер, переводящий их родной формат в ACP; клиентов около 40 — среди них Zed, JetBrains, Obsidian и Open Claw (одновременно и клиент, и агент).
- Минимальный ACP-агент — это всего четыре функции: initialize (сообщить версию протокола и возможности), authenticate (здесь не нужна, ключ берётся из env), session (создать сессию-тред с рабочей директорией) и prompt (принять блоки контента и запустить цикл вызова инструментов).
- Вывод в UI строится на session update — односторонних уведомлениях внутри сессии: agent message chunk для стриминга текста, tool call и tool call update для показа статуса и результата инструмента.
- ACP проксирует файловую систему: клиент объявляет соответствующую возможность, и агент читает или пишет файлы через протокол, а не напрямую — так редактор отдаёт даже несохранённые изменения из буфера.
- Diff — отдельный тип контента: агент шлёт старый и новый текст, а Zed сам строит наглядное различие.
- Демо-агент способен доработать сам себя: получив инструменты чтения и записи, он по промпту дописывает себе инструмент терминала (клиент может объявлять возможность создавать и вести терминалы).

## 3. Практические выводы

- Чтобы подключить агента к Zed, особый код не нужен: достаточно указать команду запуска (node и путь к файлу agent.js), и Zed общается с ним по ACP.
- Начинать стоит с четырёх обязательных функций и не заявлять лишних возможностей — минимальный агент не поддерживает ничего сверх необходимого.
- Для отладки полезен встроенный в Zed ACP debug view: видно session new, ответ с session ID, prompt и stop reason — то есть реальный обмен сообщениями.
- Стриминг подключается через реакцию на text-события Anthropic SDK: каждый чанк оборачивается в session update типа agent message chunk с привязкой к session ID.
- Файловые операции лучше вести через read text file по ACP, а не через нативный fs.readFile, чтобы агент видел актуальное состояние буфера редактора.
- Код из демо сгенерирован агентом и явно не годится для продакшена — спикер прямо об этом предупреждает.

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы

- Zed — редактор кода на Rust; Agent Client Protocol (ACP), сайт agentclientprotocol.com.
- MCP и LSP — как родственные протоколы-аналоги.
- Claude Code, Codex, Gemini CLI — примеры CLI-агентов от провайдеров моделей.
- opencode и Cursor — со встроенным режимом ACP; Open Claw, JetBrains, Obsidian — среди ~40 клиентов.
- Anthropic API и TypeScript SDK — модель Opus, стриминг через text-события, ключ из env-переменной.
- Транспорт через стандартный ввод-вывод (standard IO); удалённый транспорт в работе (упомянуты люди из JetBrains).

## 5. Что полезно разработчику

- Пошаговый, воспроизводимый рецепт совместимого с ACP агента: минимальный набор функций, порядок их реализации и как это подцепить в редактор.
- Наглядно показан цикл агента: стейтлес-вызовы модели, разбор ответа на end turn или вызов инструмента, локальное исполнение read/edit и отправка результата обратно в модель.
- Понимание, как события протокола (session update, tool call, tool call update, diff) ложатся на элементы UI редактора.
- Честный показ живых сбоев (дублирование событий, обрыв соединения) — реалистичное ожидание от сырого протокола и SDK.

## 6. Что можно применить в агентных продуктах

- Разнести клиент и агент через один протокол, чтобы не писать попарные интеграции под каждый агент и каждый редактор.
- Строить вывод на односторонних уведомлениях (session update), которые агент шлёт в любой момент, отдельно от схемы «запрос-ответ».
- Показывать вызовы инструментов двухфазно: сначала tool call со статусом «в процессе», потом tool call update с результатом.
- Проксировать файловую систему через протокол, чтобы агент работал с актуальным состоянием клиента, включая несохранённые буферы.
- Объявлять возможности (capabilities) с обеих сторон — например, создание терминала на стороне клиента — и позволять агенту расширять себя новыми инструментами.

## 7. Стоит ли смотреть полностью: yes — плотное живое демо с реальным кодом и сбоями показывает ACP изнутри лучше, чем документация; ценно, если делаешь интеграцию агента с редактором.
