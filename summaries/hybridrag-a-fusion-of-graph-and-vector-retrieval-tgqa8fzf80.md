## 1. О чём доклад
Практический обзор построения гибридной RAG-системы (HybridRAG), которая объединяет граф знаний и семантический векторный поиск, с разбором этапов обработки данных, генерации триплетов, оценки качества и ускорения графовых операций.

## 2. Главные идеи
- Граф знаний фиксирует связи между сущностями гораздо детальнее, чем векторный поиск, поэтому особенно ценен для структурированных данных (ритейл, FSI, HR) или когда из неструктурированных документов можно извлечь качественные отношения.
- Качество графа полностью зависит от качества онтологии и промпт-инжиниринга при извлечении триплетов с помощью LLM — на это уходит до 80 % времени проекта.
- Глубина обхода графа (multi‑hop) улучшает контекст, но увеличивает задержку; нужен баланс, который можно сдвинуть через GPU-ускорение (cuGraph в networkX).
- Для оценки RAG-пайплайна end‑to‑end используется Ragas (faithfulness, relevancy, precision, recall и др.) или специальная reward‑модель (Llama‑2‑based 340M), обученная судить ответы других LLM.
- Тонкая настройка (LoRA) даже небольшой LLM (Llama 3.1) и аккуратная очистка документов (удаление апострофов, лишних символов) могут поднять точность извлечения триплетов с ~71 % до 87 % на тестовых 100 документах.

## 3. Практические выводы
- Начинать с чёткого определения онтологии для целевой предметной области, итеративно улучшая промпт на извлечение триплетов.
- Использовать LoRA-файнтюнинг компактных LLM для повышения качества графа; предварительно очищать документы от мусорных символов.
- Выбирать стратегию обхода графа (single/double/multi‑hop) с учётом latency‑бюджета; ускорять поиск через cuGraph (уже интегрирован в networkX).
- Оценивать весь пайплайн (retrieval + генерацию) библиотекой Ragas, при необходимости подключая собственную LLM как судью.
- Выбор подхода (семантический, графовый, гибридный) определять по структуре данных и потребности в сложных связях; для слабоструктурированных данных, плохо превращаемых в граф, графовый RAG может быть избыточен.

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы
- **cuGraph** — GPU-ускоренная графовая библиотека, доступная через networkX
- **networkX** — Python-библиотека для работы с графами, с возможностью ускорения
- **Ragas** — библиотека для end‑to‑end оценки RAG (faithfulness, answer relevancy, precision, recall и др.)
- **Llama 3.3, 3.2, 3.1** — LLM от Meta, использованные для извлечения триплетов и файнтюнинга (LoRA)
- **Lanimotron 340M reward model** — модель для оценки ответов других LLM по нескольким параметрам
- **NVIDIA** — компания спикера; предоставляет cuGraph, ускорение и референсные ноутбуки
- **Neo4j** — упомянут как стенд для обсуждения графовых технологий на конференции
- **HybridRAG** — подход, объединяющий графовый и векторный поиск
- **LoRA** — метод эффективного дообучения LLM

## 5. Что полезно разработчику
- Пошаговый рецепт создания гибридного RAG: от обработки документов, через онтологию и промпт-инжиниринг, до извлечения триплетов и оценки.
- Конкретные метрики улучшения качества: файнтюнинг + очистка данных подняли точность извлечения с 71 % до 87 %.
- Демонстрация ускорения графовых операций через cuGraph в networkX без смены привычного API.
- Примеры использования Ragas для отладки retrieval и генерации, включая возможность подставить свою LLM.
- Ссылка на публичный GitHub-репозиторий с полным кодом, ноутбуками и двухчасовым воркшопом.

## 6. Что можно применить в агентных продуктах
- Графовый компонент HybridRAG позволяет агенту глубоко понимать связи между сущностями (multi‑hop reasoning) для сложных запросов.
- Стратегии обхода графа можно адаптировать под требования по latency и depth, что критично для интерактивных агентов.
- Ragas‑оценка даёт возможность автоматически проверять faithfulness и релевантность ответов агента.
- Принцип «clean data + fine‑tuned LLM = качественные триплеты» напрямую повышает надёжность фактов, извлекаемых агентом из корпоративных документов.
