## 1. О чём доклад
Доклад о стадиях зрелости при построении системы оценки (evals) для AI-агентов. Фил Хетцел (Braintrust) делится наблюдениями о том, как команды эволюционируют от ручных проверок до автоматизированного контроля качества с учётом сложных сценариев и внешних систем.

## 2. Главные идеи
- Evals — это не юнит-тесты: они не могут быть исчерпывающими, важнее фокусироваться на конкретных видах отказов (failure modes).
- Результаты evals не обязаны быть идеальными; допустимы направленные оценки, особенно при использовании LLM as judge.
- Зрелость измеряется континуумом: от vibe checks к автоматизированным пайплайнам, работающим с производственными данными.
- Более сложный агент (тул-коллы, запись в базы, MCP) создаёт больше точек отказа, требуя оценки всей трассировки.
- Ключевой цикл — «маховик» (flywheel): сбор продакшн-трейсов, анализ ошибок, перенос примеров в офлайн-датасет и улучшение агента на основе эвалов.
- LLM-судью тоже нужно оценивать — слепая вера в него опасна.

## 3. Практические выводы
- Начинать можно с ручных оценок (vibe check) при условии обязательной документации: не только «хорошо/плохо», но и обоснование выбора.
- По мере выявления провалов переходить к автоматическим LLM-судьям, строя их на основе тех же обоснований.
- В датасет для эвалов добавлять реальные трейсы из продакшна, чтобы повысить уверенность в работе агента.
- При работе с внешними системами стараться хранить слепок состояния прямо в трассировках (snapshot) или использовать запросы с timestamp к БД, чтобы воспроизводить контекст, в котором возникла ошибка.
- Использовать topic modeling для автоматического обнаружения новых failure modes в продакшне.

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы
- Braintrust — платформа для качества агентов, объединяющая evals и observability.
- LLM as judge (LLM в роли судьи).
- Подходы: человеческая аннотация с обоснованием; код-грейдеры для детерминированных проверок; топик-моделинг; flywheel (переиспользование продакшн-трейсов в офлайн-эвалах).
- Упоминаются cursor, cloud code, codex в контексте извлечения failure modes, а также датасеты эталонных ответов (ground truth).

## 5. Что полезно разработчику
- Начинать с простого: ручная оценка 10 примеров с обязательной фиксацией причин — это нормальный первый шаг.
- Даже на раннем этапе не давать аннотаторам универсальный интерфейс; создавать видовые формы, заточенные под специфику агента.
- Автоматизируя оценку через LLM, всегда проверять согласованность судьи с эталонными человеческими решениями.
- Использовать объективные (кодовые) проверки, когда это возможно: слишком много токенов, слишком много вызовов инструментов и т.п.
- При невозможности трогать боевые системы применять моки API и инжектить состояние через трассировку.

## 6. Что можно применить в агентных продуктах
- Встроить петлю обратной связи: трейсы из боя → выявление ошибок (человек или тематическое моделирование) → пополнение офлайн-датасета → автоматические эвалы → целенаправленные улучшения.
- Управлять рисками (репутационными, финансовыми, комплаенс) через итеративное повышение качества, а не разовое тестирование.
- Масштабировать экспертизу: знания предметного эксперта из аннотаций автоматически материализуются в оценочные скрипты.
