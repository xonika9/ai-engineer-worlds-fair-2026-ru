## 1. О чём доклад

По мере роста горизонта автономной работы моделей асинхронные агенты наконец стали удобными на практике. Спикер разбирает четыре принципа, лежащие в основе нового API Managed Agents и асинхронных агентов вообще: разделение «мозга» и «рук», использование отдельного верификатора, самообучение через память и «сны», а также организационные harness'ы. Каждый принцип он подкрепляет примерами — от бенчмарка parameter golf до партий в Pokémon с Claude.

## 2. Главные идеи

- Горизонт задач вырос от 10–20 минут во времена Opus 3 до многочасовой автономной работы; вместе с ним сместились продуктовые поверхности (автодополнение и чат → синхронные, а затем асинхронные агенты) и API (Messages API → Agent SDK → Managed Agents).
- Разделение «мозга» и «рук»: harness становится stateless-процессом, работающим с сессией — append-only журналом событий, а «руки» это контейнеры-песочницы; учётные данные хранятся в отдельном хранилище, а не в песочнице.
- Неизменяемая append-only сессия работает как внешний объект контекста, к которому модель может обращаться за старым контекстом — это удобно для управления контекстом и компакции (перекликается с recursive language models).
- Верификацию стоит выносить в отдельное контекстное окно: если одна и та же история и делает работу, и оценивает её, растут конфабуляции; связка «агент-строитель + агент-верификатор» крутит цикл, пока не выполнена цель или rubric.
- Память: как у человека есть быстрая запись (гиппокамп) и офлайн-консолидация во сне (кора), так и Claude пишет память «на лету» в файловую директорию, а «сны» офлайн исправляют накопленные ошибки.
- Более мощные модели лучше пишут память не потому, что запоминают факты, а потому что лучше выбирают обобщающую абстракцию, полезную в будущих сессиях.

## 3. Практические выводы

- Не совмещать в одном контейнере агента и секреты: при долгих прогонах падение контейнера или harness'а не должно терять сессию, а модель не должна иметь прямой доступ к учётным данным.
- Строить циклы вокруг измеримого конечного состояния: в Claude Code это «goal», в Managed Agents — «outcomes»; выход из цикла только по вердикту независимого верификатора.
- Зашивать сигнал коррекции в среду, а не в человека: тогда модель сама исправляется по обратной связи (пример parameter golf, где прогон шёл до выполнения заданных критериев).
- Давать модели общий, программируемый субстрат памяти (файловая система или БД) с простыми примитивами; не навязывать жёсткую схему памяти — по «горькому уроку» модель структурирует свою память лучше, чем предписанные типы.

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы

- Claude, Managed Agents, Agent SDK, Messages API, Claude Code, Claude Tag (доступ через Slack).
- Метрика горизонта задач METR; бенчмарки parameter golf (от OpenAI), Continual Learning Bench.
- Recursive language models как рамка для работы с внешним объектом контекста.
- Модели: Opus 3, Sonnet 3.5, поколение 4.6/4.7, упомянут Codex как пример фронтир-моделей в режиме 12+ часов (названия версий по расшифровке могут быть неточны).
- Аналогия из нейробиологии: гиппокамп и кора, «сны» как офлайн-консолидация памяти.

## 5. Что полезно разработчику

- Понимание, почему асинхронный опыт был плохим, пока модели умели работать лишь час, и что именно рост горизонта задач его разблокировал.
- Приём с отдельным контекстом-верификатором против самопроверки в одном окне — общий тренд, работающий с высокоёмкими моделями.
- Наблюдение, что качество памяти растёт с ёмкостью модели за счёт лучшей дистилляции обобщений, а «сны» офлайн исправляют локально-оптимальные и ошибочные записи (пример с ловушкой в Pokémon, 5 из 5 повторов).

## 6. Что можно применить в агентных продуктах

- Проектировать архитектуру «stateless harness + append-only сессия + отдельное хранилище секретов + контейнеры-руки» для надёжных и безопасных долгих задач.
- Внедрять петли «строитель/верификатор» с измеримым конечным состоянием как общий примитив для длительной асинхронной работы.
- Давать агенту простые инструменты записи в общий субстрат памяти и добавлять офлайн-процесс консолидации; проверять его пользу отдельными оценками (eval), чтобы оправдать офлайн-вычисления.
- Рассматривать организационные harness'ы (мультиплеер, своя identity, доступ к орг-контексту, проактивность) как способ выровнять стартовые условия для всей команды.
