## 1. О чём доклад
Практический разбор того, как строить оценки (evals) для LLM‑приложений, которые реально работают в проде. Фокус — на task evals (оценка работы приложения), а не на model evals (сравнение моделей). Докладчица на примере e‑commerce чат‑бота с роутером и function calling показывает, как выстроить многоуровневые evals, почему объяснения важнее голых чисел и какие нюансы retrieval‑augmented generation влияют на качество.

## 2. Главные идеи
- Для прикладной разработки критичны task evals, а не рейтинги моделей с Hugging Face.
- Сложные приложения с роутерами требуют eval на разных уровнях: router eval (правильность ветки), component‑level eval внутри веток.
- Evals с объяснениями — лучший формат для быстрого поиска и исправления ошибок.
- Числовые скоринговые оценки ненадёжны: LLM склонны давать бинарные значения (0 или 10) вместо гранулярных.
- В RAG позиция факта в контекстном окне критична: факты, помещённые в начало длинного контекста, часто не извлекаются.
- Для генерации после retrieval помогает инструкция «сначала объясни, потом ответь» — особенно моделям, склонным к лаконичности.

## 3. Практические выводы
- Встраивайте evals с объяснениями в свои prompt‑шаблоны, чтобы получать actionable обратную связь.
- Не полагайтесь только на числовые оценки — дополните их категориальными или пояснительными.
- В RAG-пайплайнах избегайте размещения ключевых фактов в самом начале контекстного окна; тестируйте размещение.
- Если модель плохо генерирует ответ после retrieval, добавьте в промпт требование «explain yourself then answer».
- Стройте evals итеративно: от бенчмаркинга при разработке до мониторинга в проде.

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы
- Arise – компания автора, LLM evals и observability.
- Phoenix – open‑source инструмент для трейсинга и evals (показан в демо).
- Hugging Face Open LLM Leaderboard, MMLU, needle‑in‑a‑haystack test.
- Подход LLM-as‑a‑judge.
- Function calling, router‑based applications.
- Модели: GPT‑4, Anthropic Claude 2.1, Mistral.
- Событие Arise Observe (11 июля) с участием исследователей из OpenAI, Anthropic, Mistral.

## 5. Что полезно разработчику
- Многоуровневые evals (роутер, компоненты, сессия) для быстрой локализации ошибок.
- При function calling проверять не только выбор функции, но и корректность извлечённых параметров.
- Использовать трассировку и eval с объяснениями как основной цикл итеративной доработки.
- Учитывать, что порядок подачи контекста в RAG влияет на качество ответов.
- При проблемах с генерацией пробовать промпт‑инструкцию «объясни свои рассуждения, затем дай ответ».

## 6. Что можно применить в агентных продуктах
- Обязательно вводить router evals для проверки корректности выбора агентной ветки и параметров вызова.
- Многоуровневые evals позволяют изолировать сбои в длинных цепочках агентов.
- Трассировка и пояснительные оценки ускоряют итеративную доводку агентных сценариев.
