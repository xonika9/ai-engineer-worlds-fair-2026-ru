## 1. О чём доклад
Доклад о создании доменно-специфичной системы оценки LLM на примере AI-ассистента Lucy для CRM‑платформы Rechat (недвижимость). Спикеры (Emil Sedgh, CTO Rechat, и Hamel Husain) рассказывают, как прошли путь от прототипа на «ощущениях» к надёжному production‑решению с фреймворком оценок.

## 2. Главные идеи
- Начинать не с инструментов, а с юнит‑тестов и ассертов на основе реальных сбоев.
- Обязательно смотреть на данные: *“you need to look at your data and you need to fight as hard as you can to remove all friction”*.
- Логировать трейсы, проводить человеческую разметку и строить простые кастомные приложения для просмотра, чтобы убрать трение.
- Генерировать синтетические входные данные с помощью LLM для покрытия сценариев.
- Итеративно улучшать промпты, проверяя, насколько система оценки фиксирует прогресс.
- Не переходить рано к LM‑as‑a‑judge; сначала согласовать судью с человеком.
- Каркас оценок даёт почти бесплатную возможность курировать данные для файн‑тюнинга.

## 3. Практические выводы
- Сначала используйте то, что уже есть (CI, свои БД, Metabase), потом подбирайте инструменты.
- Пишите доменные ассерты (отсутствие невалидных плейсхолдеров, повторяющихся деталей, корректность рассылок и т.п.).
- Создавайте своё приложение для аннотирования, если готовые инструменты создают трение (Gradio/Streamlit/Shiny for Python).
- Ошибка — начинать с готовых метрик «токсичности» или «краткости»; фокусируйтесь на специфичных для вашей задачи.
- *“If you’re having a conversation about evals and the first thing you start thinking about is tools, that’s a smell that you’re not going to be successful.”* Сначала процесс, потом инструменты.

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы
- GPT‑3.5, React (прототип); LangSmith для трейсинга.
- Metabase для визуализации результатов ассертов.
- Gradio, Streamlit, Shiny for Python для собственных приложений просмотра/разметки.
- Синтетическая генерация входных данных через LLM.
- Файн‑тюнинг (для сложных команд, смешанного вывода и запросов обратной связи).
- Rechat (компания, продукт), AI‑ассистент Lucy.

## 5. Что полезно разработчику
- Системный подход: юнит‑тесты + логирование + разметка → цикл улучшения.
- Смотреть на данные обязательно; автоматизация без взгляда на реальные ошибки не работает.
- Промпт‑инжиниринг — быстрый способ проверить зрелость eval-системы.
- Оценки для файн‑тюнинга: хорошие кейсы фильтруются через eval, плохие идут в доработку.

## 6. Что можно применить в агентных продуктах
- Тот же фреймворк для сложных мультитуловых команд: генерировать синтетические запросы агентам, размечать трассы, проверять корректность вызовов.
- Ассерты под специфику агента (структурированный+неструктурированный вывод, запрос уточнений, каскадные тул‑вызовы).
- Постепенное накопление размеченных данных для файн‑тюнинга модели агента.
