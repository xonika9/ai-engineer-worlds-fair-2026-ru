## 1. О чём доклад
Михаил Хаблих (продакт-менеджер Chrome DevTools, Google) делится инженерными уроками, полученными при создании Chrome DevTools for Agents — MCP‑сервера, который даёт агентам возможность отлаживать, профилировать и улучшать веб-страницы, вместо того чтобы кодить вслепую.

## 2. Главные идеи
Агенты — отдельный класс пользователей: у них общие с людьми цели, но разные когнитивные «узкие места» (токены вместо визуальной сложности). Интерфейс для агента нужно проектировать с учётом эффективности, обнаружимости, безопасности и устойчивости к ошибкам. Четыре ключевых вывода: измерять топливную эффективность через tokens per successful outcome, превращать ошибки в playbooks восстановления, аудит описаний инструментов на «запахи качества» и никогда не жертвовать доверием ради удобства.

## 3. Практические выводы
- **Tokens per successful outcome** — метрика эффективности интерфейса; считать глобально нельзя, только в разрезе конкретных пользовательских сценариев.
- **Token burn** снижается тремя способами: категоризация инструментов (скрытие нишевых), slim‑режим (всего три инструмента), CLI‑интерфейс для постобработки на стороне агента.
- **Error recovery**: добавлять полезные сообщения, проактивные «объезды» (detours), диагностические skills для самовосстановления. Ошибки стоят токенов.
- **Discoverability**: отказались от монолитного `debug webpage`, разложили на 25 инструментов. 97% MCP‑описаний имеют «запахи качества» — схема и есть UI для агента. Нужны чёткие цель, критерии активации и минимально жизнеспособное описание. Skills полезны, но не бесплатны — могут вызываться не по делу.
- **Доверие**: Auto‑connect удобен, но фрикцион необходим — пользователь должен осознанно давать согласие. Три тира безопасности: локальная разработка (human‑in‑the‑loop), CI‑окружение (контейнеры, отдельные профили), агенты с полным доступом в интернет (YOLO‑режим, нужны allow‑list и защита от prompt injection).

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы
- Chrome DevTools for Agents (MCP‑сервер)
- Gemini CLI, Claude Code, Codex, OpenClaw
- Lighthouse, Core Web Vitals (LCP, INP, CLS)
- Slim mode, tool categorization, CLI interface
- Diagnostic playbooks (skills)
- Летальный три‑фактор (Simon Willison)
- MCP tool description quality paper (97% smell score)

## 5. Что полезно разработчику
- Не скармливать агенту сырые trace‑файлы — давать семантические сводки в markdown.
- Начать измерять tokens per successful outcome даже с несовершенной метрикой: это лучше интуитивных решений.
- Обязательно добавить в инструменты осмысленные ошибки с подсказками для самовосстановления.
- Пересмотреть описания инструментов: объяснять, *когда* применять, а не только *что* делает инструмент.

## 6. Что можно применить в агентных продуктах
- Сегментировать инструменты по частотности и включать их через флаги или slim‑режимы, чтобы беречь контекстное окно.
- Предоставлять CLI‑обвязку для цепочек команд и постобработки без участия модели.
- Создавать skill‑плейбуки для типовых отказов (например, проблемы при настройке MCP).
- Явно проектировать доверенные границы: автоподключение помнить опасно, особенно при делегировании агенту доступа к реальному браузеру.
