## 1. О чём доклад
Доклад о том, как команда LangFuse создавала скилл для кодинг-агентов, чтобы автоматизировать добавление трассировки, оценки и управления промптами в проекты пользователей. Марк Клинген делится уроками, полученными при «обучении» агентов правильно интегрировать LangFuse в разные приложения, и размышляет о роли скиллов как формализованных инструкций для повышения надёжности агентов по сравнению с жёсткими воркфлоу.

## 2. Главные идеи
- Скиллы — это способ дать агенту формализованный ярлык, чтобы он работал надёжнее, чем полностью автономный агент, но гибче, чем фиксированный воркфлоу.
- Основные проблемы при добавлении LangFuse «из коробки»: устаревшие данные в предобученном контексте, неоптимальная настройка и лишние шаги, когда агент сначала ошибается, а потом исправляет ошибки.
- Наблюдение за трассами выполнения (tracing) даёт 80% инсайтов для улучшения скилла и обнаружения неожиданных сценариев использования.
- Агенту нужно помогать навигировать в большом объёме документации: через sitemap, поисковый эндпоинт с retrieval-архитектурой и явные подсказки о том, как запрашивать markdown.
- Целевая функция (как измерять успех) критична: неправильно заданная цель приводит к удалению нужных шагов (например, запрос подтверждения у пользователя или версионирование промптов).

## 3. Практические выводы
- Смотреть трассы агента — самое быстрое вложение времени для понимания его реального поведения и ошибок.
- Агентам не лень передавать лишние переменные окружения; можно запрашивать у пользователя нужные параметры (например, data region) вместо умолчаний.
- CLI с флагом `--help` снижает галлюцинации параметров — агент сразу видит реальный интерфейс.
- Даже простой базовый набор оценок (eval) лучше его отсутствия; можно начать с 5 типовых сценариев.
- Динамический контент (документацию) лучше давать по ссылке, а не копировать в скилл — иначе он устаревает.
- Автоисследование с оценкой целевой функции помогает генерировать идеи для улучшения скилла, но требует ручного ревью.

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы
- LangFuse (open-source платформа трейсинга и оценок для LLM-приложений)
- Cloud Code (кодинг-агент из доклада)
- LangChain
- OpenAI
- Anthropic
- Cursor
- Подходы: tracing/observability, LLM-as-a-judge, online/offline evals, human-in-the-loop, prompt management, RAG-based search endpoint, skills как формализованные подсказки агенту.

## 5. Что полезно разработчику
- Поисковый эндпоинт для документации позволяет и продукту отвечать на естественно-языковые вопросы, и команде отслеживать проблемы пользователей через логи поисков.
- Оборачивание API в CLI даёт агенту возможность выполнять всё, что раньше делалось вручную в UI (например, настройку оценок).
- Не стоит дублировать контент документации внутрь скилла — лучше ссылаться на живые источники и указывать дату, чтобы агент определял актуальность.
- Использовать трассировку собственного процесса разработки скилла — «ешь свою собачью еду» — чтобы видеть ошибки и медленные маршруты.

## 6. Что можно применить в агентных продуктах
- Скилл должен просить агента задавать уточняющие вопросы пользователю, прежде чем принимать решения о настройке.
- Карта документации (sitemap) и хинт про заголовок `Accept: text/markdown` экономят токены и ускоряют навигацию.
- Простые eval-чеки на естественном языке (например, «в трейсе должны быть retrieval-спаны») позволяют быстро обнаруживать регрессии при итерациях.
- Механизм timestamp в скилле и проверка «старше месяца — обновить» решает проблему устаревания.
- Автоисследование: агент-оптимизатор пробует множество паттернов и предлагает изменения, которые затем проходят человеческий фильтр.

## 7. Стоит ли смотреть полностью: yes
Доклад содержит конкретные инженерные уроки на реальном продукте, балансирует между концептуальными размышлениями и практическими хаками — полезно для команд, строящих инфраструктуру для агентов или внедряющих coding agents в CI/CD.
