## 1. О чём доклад
О практическом внедрении больших языковых моделей в разработку внутри Jane Street — компании, где почти всё пишут на редком функциональном языке OCaml. Спикер рассказывает, как они строят собственные дообученные модели для генерации кода, создают сервис для оценки и RL‑тренировки, и унифицируют редакторные интеграции под Emacs, VS Code и Neovim через sidecar‑архитектуру, не зависящую от готового инструментария.

## 2. Главные идеи
- Готовые AI‑инструменты плохо работают с OCaml, потому что моделям не хватает обучающих данных, а внутренние системы (сборка, код‑ревью, репозиторий) нестандартны.
- Чтобы получить полезную модель, данные для обучения должны повторять форму целевого запроса: контекст → промпт → дифф.
- Вместо полных пулл‑реквестов или коммитов используют снапшоты рабочих станций каждые 20 секунд. Переходы «зелёный → красный → зелёный» в статусе сборки дают изолированные изменения и исправления ошибок.
- Описания к изменениям генерирует LLM, затем их сжимают до уровня, похожего на человеческий запрос.
- Основной контур обучения: supervised fine‑tuning на собранных диффах, затем reinforcement learning на верифицируемых задачах — код должен парситься, проходить проверку типов и тесты.
- Для RL и оценки создан Code Evaluation Service (CES): предварительно «прогретая» сборка принимает диффы модели и немедленно сообщает, «зелёный» результат или «красный».

## 3. Практические выводы
- Без осмысленной оценки модели могут давать катастрофически смешные результаты (например, модель код‑ревью ответила «сделаю завтра», обучаясь на репликах людей).
- Тренировка на снапшотах рабочего окружения позволяет собирать данные, максимально приближенные к реальному рабочему процессу, без ручной разметки.
- Архитектура sidecar (AID) на машине разработчика окупается: смена моделей, стратегий построения контекста и инструментов происходит в одном месте и действует сразу во всех редакторах.
- A/B‑тестирование моделей по acceptance rate помогает принимать решения на реальных метриках, а не на ощущениях.

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы
- OCaml, Iron (внутренняя система код‑ревью), Mercurial, монорепозиторий.
- Emacs (самый популярный редактор), VS Code, Neovim.
- AID (AI Development Environment) — sidecar‑сервис для промптов, контекста и взаимодействия с LLM.
- Code Evaluation Service (CES) — сервис быстрой проверки диффов на «живой» сборке.
- Статья CodeCompose от Meta (дообучение модели для языка Hack, который написан на OCaml).

## 5. Что полезно разработчику
- AID как sidecar позволяет перезапускать сервис без перезапуска редактора и мгновенно доставлять обновления.
- Тонкие интеграции в редакторы: в VS Code — визуальный сайдбар, в Emacs — работа внутри markdown‑буфера с привычными текстовыми командами.
- Подключаемые доменно‑специфичные инструменты разных команд становятся доступны сразу во всех редакторах без дублирования кода.

## 6. Что можно применить в агентных продуктах
- Принцип pluggable архитектуры: единый слой для моделей, промптов и контекста, к которому подключаются разные агенты или интерфейсы.
- Подход к сбору обучающих данных через снапшоты окружения и сигналы сборки применим для тренировки агентов, которые исправляют ошибки или генерируют изменения в больших кодовых базах.
- RL с автоматической верификацией (компиляция/тесты) — готовый шаблон для обучения агентов, действующих в闭环 (closed loop) с реальным кодом.
