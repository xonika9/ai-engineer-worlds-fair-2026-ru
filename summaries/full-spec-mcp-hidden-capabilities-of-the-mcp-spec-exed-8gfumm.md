## 1. О чём доклад  
Доклад о скрытых возможностях полной спецификации Model Context Protocol (MCP), которые редко используются разработчиками. Спикер из Microsoft/VSCode показывает, как выйти за пределы «обёрток над API» и реализовать богатые, stateful-взаимодействия агентов, уже заложенные в спецификации. Основной посыл — экосистема должна догонять спецификацию.

## 2. Главные идеи  
- Существует разрыв между полнотой спецификации MCP и реальными имплементациями: *«MCP is just another API wrapper syndrome… people just want to ship»*.  
- Большинство использует только инструменты (tools), игнорируя ресурсы, подсказки и сэмплинг.  
- Динамическое обнаружение (dynamic discovery) и ресурсы (resources) позволяют серверам менять набор инструментов на лету и передавать клиенту состояние, например, файлы или скриншоты.  
- Сэмплинг (sampling) — это возможность сервера запрашивать LLM-завершения у клиента; в VS Code Insiders он уже реализован.  
- Улучшение developer experience критически важно: появился dev mode с отладкой прямо в VS Code.  
- Необходимо тестировать draft-фичи спецификации (streamable HTTP, elicitations) и давать обратную связь, чтобы они стали стабильными.  

## 3. Практические выводы  
- Для повышения качества агентов используйте полную спецификацию, а не только инструменты.  
- Применяйте динамическое обнаружение, чтобы показывать ИИ-модели только релевантные инструменты в конкретный момент.  
- Реализуйте streamable HTTP вместо SSE для снижения нагрузки на серверы.  
- Используйте будущую фичу elicitations для прямого запроса информации у пользователя из инструмента.  
- Тестируйте draft-спецификации (OAuth, streamable HTTP) и оставляйте обратную связь разработчикам SDK и клиентов.  

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы  
- **Клиенты/платформы:** VS Code, VS Code Insiders.  
- **Серверы/примеры:** GitHub MUD MCP (демо dungeon crawler), Playwright (скриншоты).  
- **Спецификации/концепты:** MCP Spec (tools, resources, prompts, sampling), dynamic discovery, streamable HTTP, elicitations, OAuth spec, Community Registry.  
- **Инструменты разработчика:** VS Code dev mode, отладчик для Python и Node.js.  
- **Модели/фреймворки:** GPT-4.1, LangChain (исследование проблем качества при большом числе инструментов).  

## 5. Что полезно разработчику  
- **Dev mode в VS Code:** встроенная консоль и отладка MCP-серверов — можно ставить точки останова и проходить код.  
- **Динамическое обнаружение:** возможность менять список инструментов «на лету» в зависимости от контекста, снижая путаницу у LLM.  
- **Сэмплинг:** прогрессивное улучшение, когда сервер может попросить LLM-клиента что-то отформатировать или обобщить, вместо того чтобы отдавать сырые данные.  
- **Контроль инструментов:** в VS Code появились выбор инструментов на чат, упоминание инструментов в промпте и пользовательские наборы инструментов (tool sets).  

## 6. Что можно применить в агентных продуктах  
- **Ресурсы (resources):** идеальны для передачи артефактов (скриншоты, сгенерированные файлы) обратно пользователю и LLM через семантический слой.  
- **Чтение окружения через ресурсы и roots:** агент может сам изучить структуру папок, установленные пакеты и определить, с каким фреймворком работает пользователь, прекращая задавать лишние вопросы.  
- **Stateful-инструменты:** реализация паттернов вроде «баттл с монстром» в игре, где инструмент атаки появляется только при наличии цели.  
- **Elicitations (draft):** механизм, позволяющий инструменту запросить конкретные данные напрямую у пользователя, не полагаясь только на интерпретацию LLM.  

## 7. Стоит ли смотреть полностью: yes / maybe / no  
**yes** — особенно если вы разрабатываете MCP-серверы и хотите использовать возможности за пределами инструментов, отлаживать их эффективнее и понимать вектор развития спецификации.
