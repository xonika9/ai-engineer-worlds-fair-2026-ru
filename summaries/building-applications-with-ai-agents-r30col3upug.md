## 1. О чём доклад
Доклад посвящён строительству надёжных приложений на базе ИИ-агентов. Спикер делится опытом создания Microsoft Security Copilot и материалами своей будущей книги. Рассматриваются ключевые компоненты продакшн-систем: использование инструментов (tools), паттерны оркестрации, подходы к оценке качества (evaluation) и наблюдаемости (observability). Особый акцент сделан на разрыве между быстрым прототипом с точностью 70% и сложностью доведения систем до промышленной надёжности.

## 2. Главные идеи
- **Агентность — это спектр, а не бинарное свойство.** Роботизированная автоматизация (RPA) имеет низкую агентность, но высокую эффективность. При добавлении агентности важно не терять эффективность, иначе можно скатиться к «плохим чат-ботам».
- **Ловушка прототипа.** Легко достичь 70% точности, но невероятно трудно закрыть длинный хвост сложных сценариев на пути к 90%+.
- **Меньше инструментов — выше точность.** Десятки и сотни API не должны превращаться в сотни инструментов агента: семантические коллизии снижают точность модели.
- **Детерминированная бизнес-логика.** Критически важные условия не стоит доверять большой языковой модели; их следует выносить в фиксированный код, а агенту оставлять обновление состояний и поиск информации.
- **Безопасность — новый класс уязвимостей.** Агенты требуют эшелонированной защиты и «предохранителей» (trip wires), чтобы в критических случаях передавать управление человеку.

## 3. Практические выводы
- Начинайте проектирование с оценки качества (evaluation), а не с кода агента. Это аналог test-driven development.
- Группируйте инструменты в семантически близкие кластеры и ограничивайте число доступных агенту инструментов за один вызов.
- Если цепочки вызовов становятся слишком сложными — переходите к более агентному паттерну (автономный цикл принятия решений моделью).
- Используйте синтетические данные для затравки eval-сетов, если нет доступа к реальным данным пользователей.
- Инвестируйте в Observability (OpenTelemetry), кластеризацию ошибок и автоматическую суммаризацию сбоев.

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы
- **Microsoft Security Copilot** — пример индустриального агентного продукта.
- **Intel Agent** — open-source библиотека для генерации синтетических входных данных.
- **PyRIT** (Microsoft) — open-source инструмент для red-teaming и проверки устойчивости агентов.
- **Label Studio** — фреймворк для создания размененных датасетов оценки.
- **TextRad / DSPy** — инструменты для автоматической оптимизации промптов и гиперпараметров (градиенты через судью-модель).
- **OpenTelemetry** — стандарт трассировки и логирования.
- **Agent-to-Agent Protocol** — перспективный, но пока ранний протокол для взаимодействия агентов разных команд.

## 5. Что полезно разработчику
- Эмпирическое правило: рост числа инструментов **на один промпт** ведёт к падению общей точности. Инструмент должен описываться как одно атомарное действие, понятное человеку.
- Не перекладывайте бизнес-правила на LLM. Используйте модель для принятия решений в условиях неопределённости, а валидацию и критические условия реализуйте детерминированно.
- Трассировка через OpenTelemetry — обязательный минимум для понимания «подводной части айсберга» ошибок в проде.

## 6. Что можно применить в агентных продуктах
- Концепцию «спектра агентности» при дизайне фич: начинайте с простых цепочек (fixed chains) и переходите к автономным циклам (agentic loops) только когда детерминированная логика становится неподъёмной.
- Мультиагентную архитектуру как способ масштабирования количества скиллов: отдельные агенты для отдельных групп инструментов с единым координатором-роутером.
- Внешний цикл улучшения: автооценка с помощью модели-судьи и автоматическая оптимизация промптов на основании агрегированных ошибок.
