## 1. О чём доклад
Кейс компании PFF (спортивная аналитика, 200 сотрудников, 20 инженеров) по переходу от классического Scrum к инженерной организации, где AI-агенты берут на себя рутину, упраздняют церемонии и позволяют двум сильным инженерам за два месяца сделать то, на что раньше требовалось четыре.

## 2. Главные идеи
- Оптимизация не самого инженера, а агента: чем быстрее агент, тем выше отдача.
- Команда из двух инженеров с агентами: x25 чаще деплои, x10 продуктивность (с учётом сложности кода и задач), качество продукта по опросам — 8,6 из 10 (против прежних 7–7,5).
- Главная метрика успеха — удовлетворённость клиентов, а не количество PR или строк кода.
- Scrum-ритуалы (стендапы, планирования, ретро) исчезли за ненадобностью.
- Новый цикл: агент интервьюирует → спека → лёгкий дизайн-документ (LDD) → автоматическая генерация задач и PR без блокировок.
- QA-агент проверяет критерии приёмки, в перспективе — автоисправление дефектов.
- Инженеры разного склада: любознательные выигрывают, жёстко ориентированные на готовые спецификации — проигрывают.

## 3. Практические выводы
- Начинайте с некритичных систем и proof-of-concept, затем расширяйте область.
- Внедряйте поэтапно: сначала лучшие инженеры с глубоким знанием системы, потом остальные.
- Разбейте цикл разработки как фабрику на составные навыки (composable skills), кодируйте в них принятые паттерны и культуру.
- Убирайте процессы, не несущие реальной пользы; спрашивайте «зачем эта встреча?».
- Автоматизируйте ненавистные рутины: стилистическое код-ревью, обновление статусов задач по состоянию PR.
- Не пытайтесь обучить всех разом; не будьте чрезмерно консервативны — отставание от конкурентов нарастает экспоненциально.

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы
- PFF (спортивные данные для NFL/NCAA)
- AI-агенты (coding agents, QA agent)
- Cloud Ops (триггер экспериментов)
- Scrum (Agile-церемонии признаны избыточными)
- Trunk-based development, feature flags
- Service repository pattern (для API)
- Lightweight design documents (LDD)
- Автоматическое создание задач и PR, автообновление статусов
- Опросы удовлетворённости клиентов (статистически значимые)
- Composable skills (branch naming, feature-флаги, генерация API по паттернам)
- Агентное код-ревью для стиля и нейминга

## 5. Что полезно разработчику
- Рутина передана агентам: стилевое код-ревью, обновление трекера по статусу PR, генерация задач из спеки.
- Быстрая обратная связь: «хаддлы» раз в два дня с продуктом, дизайном, мгновенный деплой MVP в прод.
- Освобождение времени на продуктовое «чутьё» и архитектурные решения, а не на церемонии и согласования.
- Автоматическое устранение блокирующих зависимостей: система подсвечивает и разруливает последовательные задачи.
- Отсутствие посредников: нет project-менеджера, «испорченного телефона» при передаче требований.

## 6. Что можно применить в агентных продуктах
- Конвейер «агент-интервью → спека → LDD → задачи → PR» как автономный продуктостроительный поток.
- QA-агент, сверяющийся с acceptance criteria, с планами перехода к самозакрывающимся PR.
- Разделение код-ревью: агенту — стиль/именование, человеку — системный дизайн.
- «Фабричные» навыки, композируемые под уникальные правила орг-ции (trunk-based, конкретный паттерн API), чтобы избежать ощущения «сгенерировано cloud‑кодом».
- Фильтрация сторонних навыков: не брать чужие, если они навязывают архитектурные решения, конфликтующие с вашей культурой.
