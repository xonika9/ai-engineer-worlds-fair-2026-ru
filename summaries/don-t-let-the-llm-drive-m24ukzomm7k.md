1. О чём доклад  
Доклад Microsoft о создании живого голосового AI-тьютора Ace, который надёжно проводит полный урок. Главная идея: многошаговый агент ломается не из-за слабых промптов, а из-за того, что модель сама управляет процессом. Решение — «не давать LLM водить», отдавая контроль «обвязке» (harness).

2. Главные идеи  
- Надёжность — проблема контроля, а не промптинга. Модель — талант, harness — режиссёр; модель не должна помнить этапы.  
- Урок реализован как конечный автомат с состояниями (intro, teach, check, grade, advance, wrap). Каждый шаг отправляет модели жёсткий контракт: сделай одно действие и верни результат.  
- Harness валидирует ответ, меняет состояние и решает, что делать дальше. Модель никогда не определяет текущий шаг.  
- Вынос управляющей логики позволяет заменить тяжёлую модель (Opus 4.7) на лёгкую (Haiku 4.5) без потери качества — экономия денег, времени и снижение задержек.  
- Подход универсален: голосовые ассистенты, coding agents, Ops Run Books, onboarding flows.  
- Правило: если поведение агента похоже на подбрасывание монетки — забирайте у модели контрольный поток.

3. Практические выводы  
- Проектируйте агента как явный конечный автомат с чёткими переходами.  
- Каждому шагу давайте модели изолированный вход и требуйте конкретный выход.  
- Не позволяйте модели отслеживать состояние; она должна только исполнять инструкции.  
- Внедряйте harness для валидации и маршрутизации — это повышает наблюдаемость и упрощает оценку.  
- Начинайте с более лёгких моделей, если контроль жёстко вынесен вовне.

4. Упомянутые инструменты, фреймворки, компании, продукты и подходы  
- Продукт: Ace (живой AI-тьютор, Microsoft)  
- Модели: Anthropic Opus 4.7, Haiku 4.5  
- Концепция: harness engineering («инженерия обвязки»)  
- Архитектура: конечные автоматы (state machines)  
- Подход: «модель предлагает, harness решает», «let it talk, but don’t let it drive»

5. Что полезно разработчику: AI-агенты, браузерные и офисные агенты, harness, sandboxes, tool execution, developer tools, evals, observability, агентном продукте  
- Harness как способ жёсткого контроля потока прямо применим к AI-агентам, включая браузерные и офисные сценарии.  
- Изоляция шагов упрощает sandboxes и tool execution — модель получает ровно то, что нужно для конкретного действия.  
- Для developer tools: конечные автоматы и валидация выхода снижают хрупкость и облегчают отладку.  
- Observability и evals выигрывают: логи harness (как показано в видео) делают каждый шаг прозрачным и проверяемым.  
- агентном продукте может использовать эти принципы как основу для надёжных студийных агентов.

6. Что можно применить в агентных продуктах  
- Внедрить конструктор конечных автоматов для сценариев: пользователь описывает шаги, студия автоматически генерирует harness и контракты для LLM.  
- Предоставить шаблоны «контрактов» — чётких входов/выходов для типовых действий агента.  
- Использовать облегчённые модели с активной обвязкой, чтобы снизить стоимость и задержки внутри Studio.  
- Добавить визуализацию логов harness для observability прямо в интерфейсе студии.

7. Возможная идея для поста  
«Почему ваш LLM-агент сходит с ума на полпути и как harness engineering спасает ситуацию. Опыт Microsoft Ace: модель не должна водить, она должна играть свою роль по сценарию конечного автомата». Или коротко: «Don’t let the LLM drive».

8. Стоит ли смотреть полностью: yes  
Доклад короткий (~6 минут), отлично доносит ключевую идею с конкретным примером и демонстрацией логов. Рекомендуется всем, кто строит многошаговых агентов.
