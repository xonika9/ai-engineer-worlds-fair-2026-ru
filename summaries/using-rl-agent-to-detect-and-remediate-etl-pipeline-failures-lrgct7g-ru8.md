## 1. О чём доклад

RL-система для автоматического обнаружения и устранения сбоев ETL-конвейеров (AWS Glue). Агент по событиям читает логи и метаданные схем, классифицирует отказы, оценивает риск и безопасно выбирает одно из действий: retry, coerce schema, rollback, quarantine, escalate или log. Цель — не заменить инженера, а избавить его от повторяющейся ночной рутины при типовых авариях.

## 2. Главные идеи

- Жёсткое разделение: детерминированные правила — для фактов, табличный Q-learning — для контекстного выбора действий, внешний safety layer — для ограничений.
- Эскалация — не провал, а способность агента честно сказать «мне здесь нельзя действовать автоматически».
- На синтетическом бенчмарке RL не превзошёл детерминированную политику; надёжность исходит от структурированного состояния и явных ограничения.

## 3. Практические выводы

1. Измеряемое — детерминированными правилами; контекстные решения — обучением.
2. Safety‑ограничения выносить за пределы RL-политики, чтобы обновление модели не могло незаметно расширить полномочия.
3. Эскалацию и валидацию результата проектировать как полноценные исходы.
4. Оценивать на многих seeds, сравнивать с простыми базовый вариант (детерминированной и случайной стратегиями).
5. Для self-healing системы не нужна большая модель — нужны чёткое состояние, ограниченные действия и воспроизводимость.

## 4. Упомянутые инструменты, фреймворки, компании, продукты и подходы

AWS Glue, EventBridge, Lambda, CloudWatch, Glue Data Catalog, S3. Методы: табличный Q-learning, anomaly detection rules, schema profiling, drift detection, data quality analysis, error classification, risk scoring, safety layer, shadow mode deployment.

## 5. Что полезно разработчику

Паттерн «детерминированные сенсоры + обученное действие + внешняя безопасность» прямо переносим на AI-агентов (браузерных, офисных). Эскалация как штатная возможность агента при недостатке уверенности. Метрики precision/recall, success rate, non-escalation rate и обязательные repeated runs для evals. Shadow mode как безопасный путь к продакшен.

## 6. Что можно применить в агентных продуктах

Архитектурный шаблон с разделением rules / policy / ограничения для любых tool-execution агентов. Табличный Q-learning для объяснимых решений в маленьких пространствах. Синтетические бенчмарки для воспроизводимой оценки. Эскалация как встроенное действие агента и практика shadow‑деплоймента перед наделением реальными правами.

## 7. Стоит ли смотреть полностью: yes — доклад даёт инженерно‑пригодную рамку для построения надёжных агентов с воспроизводимыми экспериментами, прямо применимую к dev‑tools и агентным системам.
